# SPECCODING 开发规范文档

## 1. 概述
本规范旨在统一项目开发全生命周期的文档与测试管理，确保产品、技术、测试三部分协同一致，支持迭代演进与自动化验证。  
**技术栈限制**：
- 语言：TypeScript / JavaScript
- 桌面应用框架：Electron 38
- 前端框架：React 19
- 后端服务：独立 Hono HTTP + WebSocket 服务器（运行于 Node.js 环境）

所有开发人员必须遵守此规范，并在编码前获得设计文档的确认。

## 2. 目录结构
项目根目录需包含以下目录及文件结构（Electron 项目推荐结构）：
```plain
/
├── docs/
│ ├── product/ # 产品文档目录
│ ├── tech/ # 技术文档目录
│ └── test/ # 测试用例文档目录
├── tests/
│ └── script/ # 测试脚本目录
├── src/
└── ...
```

## 3. 产品文档规范
- **存放位置**：所有产品文档必须位于 `docs/product/` 目录下。
- **命名格式**：`iter_<迭代轮数>_<迭代内容>.md`  
  示例：`iter_1_用户认证模块.md`、`iter_2_文件传输.md`  

## 4. 技术文档规范
技术文档分为前端（React）、后端（Hono）与主进程（Electron）三部分，均存放于 `docs/tech/` 目录，并遵循迭代命名。

- **存放位置**：`docs/tech/` 目录。
- **命名格式**：`iter_<迭代轮数>_<前端|后端|主进程>_<简述>.md`  
  示例：`iter_1_frontend_页面布局.md`、`iter_1_backend_WebSocket接口.md`、`iter_1_main_窗口管理.md`  
  若前后端内容合并撰写，可使用 `iter_<迭代轮数>_<简述>.md`，但必须在文档内明确区分各部分章节。


## 5. 测试规范
测试分为测试用例文档与自动化测试脚本，分别位于 `docs/test/` 和 `tests/script/` 目录。

### 5.1 测试用例文档
- **存放位置**：`docs/test/`
- **命名建议**：`test_case_<迭代轮数>_<功能>.md` 或 `test_case_<功能描述>.md`，无强制统一，但需易于识别。

### 5.2 测试脚本
- **技术栈**：使用 TypeScript/JavaScript 测试框架，推荐 Vitest（单元/集成）、Playwright（UI 测试）或 Jest。
- **存放位置**：`tests/script/`
- **命名规则**：必须以 `test_` 开头，例如 `test_api_login.ts`、`test_websocket.ts`、`test_ui_window.ts`。
- **运行要求**：
  - 脚本必须可独立运行（无需人工交互），能够自动执行全部或指定测试。
  - 提供统一的运行入口，如 `test_runner.ts` 或 `package.json` 中的 `test` 脚本，该入口自动发现并执行所有 `test_*` 文件。
  - 脚本执行后需输出明确的测试结果（成功/失败），失败时应提供足够定位信息。
- **修正流程**：
  - 开发完成后，必须运行所有测试脚本。
  - 若存在失败的测试，开发人员须修正代码或测试脚本（视错误原因而定），直至全部通过。
  - 测试通过后方可视为当前迭代完成。

## git规范

开发新功能时，先拉取main分支，然后创建新的feature分支。
合并时，先拉取main分支，然后合并到feature分支。如果有冲突，则需自行解决。如果无法确定，在提醒人工处理。
解决冲突后，将feature分支合并到main分支。并推送。

- **分支命名**：`feature/迭代轮数_迭代内容` 或 `bugfix/迭代轮数_迭代内容`，例如 `feature/1_用户认证模块`、`bugfix/2_文件传输`。
- **提交规范**：
  - 提交前必须先运行所有测试脚本，确保代码质量。
  - 提交时，必须包含详细的变更描述和测试结果。
  - 提交后，必须运行所有测试脚本，确保所有功能正常。

# Model Operating Modes

## 目标

本文件用于统一定义：

- 不同能力模型在 `openkin` 仓库中的工作模式
- 每种模式下允许做什么、不允许做什么
- 每种模式对应的提示词模板
- 遇到什么情况必须升级到更强模型或人工

核心原则：

> 不要依赖模型“自己知道自己强或弱”，而要由外部明确告诉它当前处于哪种模式，并要求它严格按规则行事。

---

## 1. 模式总览

建议至少区分三种模式：

### 1.1 `high-capability mode`

适用场景：

- 架构设计
- 跨层 contract 收口
- 多方案权衡
- 高风险边界调整

特点：

- 可以做方向判断
- 可以定义新规则
- 可以规划较大阶段
- 可以冻结弱模型后续执行所需的单一路径决策

### 1.2 `budget mode`

适用场景：

- 低预算阶段
- 能力较弱模型
- 以稳定推进为主，不以创造性设计为主

特点：

- 只能在现有规则和计划内执行
- 以小步实现、修复、验证为主
- 遇到边界问题立即停下

### 1.3 `maintenance mode`

适用场景：

- 文档整理
- 链接修复
- 归档移动
- 脚本更新
- 低风险回归修复

特点：

- 修改范围小
- 任务边界清晰
- 更适合自动化批量推进

---

## 2. 如何选择模式

### 用 `high-capability mode`

当任务包含以下任一情况时，优先使用高能力模型：

- 需要新增或重写跨层 contract
- 需要调整架构边界
- 需要修改 SDK 对外接口
- 需要修改 channel 生命周期协议
- 需要处理多个可选架构方案
- 弱模型连续两轮无法通过 `pnpm verify`
- 需要把执行计划改写成弱模型可直接执行的工作单

### 用 `budget mode`

当任务满足以下特点时，优先使用较弱模型：

- 已经有执行计划
- 执行计划已经由高能力模式冻结关键决策
- 已经有明确验收标准
- 修改范围较小
- 默认可以通过 `pnpm verify` 自证
- 不需要做新的跨层设计决策

### 用 `maintenance mode`

当任务主要是：

- 文档治理
- 归档整理
- 链接修复
- 脚本小改
- lint 失败修复

可以使用更廉价的模型持续推进。

## 3. `high-capability mode` 提示词模板

适用于：

- 设计
- 规划
- 收口 contract
- 处理高风险变更

```text
你当前处于 high-capability mode。

你的职责是做方向判断、收口 contract、处理高风险边界，并把结果沉淀成仓库内可执行的规则、计划和文档。

你可以做的事情：
- 设计或调整跨层 contract
- 收口架构边界
- 规划下一阶段执行计划
- 定义新的自动化约束与反馈回路
- 把后续要交给弱模型的任务写成单一路径工作单

你必须遵守：
1. 先阅读 `AGENTS.md`
2. 先阅读 `../index.md`
3. 优先把判断沉淀到文档、计划或脚本，而不是只留在聊天里
4. 如果方案会影响多个层级，必须明确写出影响范围和不做什么
5. 如果开始实现，结束前必须运行 `pnpm verify`

你的输出要求：
- 先说明问题边界
- 再说明取舍依据
- 再给出最小可执行方案
- 如果涉及实现，说明如何验证
```

## 4. `budget mode` 提示词模板

适用于：

- 低预算模型
- 能力较弱模型
- 按计划执行

```text
你当前处于 budget mode。

你的职责不是重新设计系统，而是在现有仓库规则、执行计划和验证脚本约束下，完成小范围、低风险、可验证的实现任务。

你必须遵守以下规则：

一、开始前先读
- `AGENTS.md`
- `../index.md`
- 当前对应的执行计划
- 计划内已经冻结的允许修改目录、验收命令与升级条件
- 与任务直接相关的实现文件

二、你可以做的事
- 按执行计划完成小范围实现
- 修复 `pnpm verify` 失败
- 补充低风险测试、脚本和文档
- 在既有 contract 内补充实现
- 在高能力模式已冻结的 exec-plan 内，补齐已批准的 shared contract 字段、路由常量或测试入口

三、你不可以做的事
- 新增或重写跨层 contract
- 修改总体架构方向
- 修改 SDK 对外接口
- 修改 channel 生命周期协议
- 做无计划的大范围重构
- 自行扩展需求范围
- 自行新增未冻结的 endpoint、event、DTO 或跨层语义

四、每次修改后必须执行
- `pnpm verify`

五、如果出现以下情况，立即停止并汇报，不要继续尝试
- 需要改架构边界
- 需要改计划外的共享 contract
- 连续两轮无法通过 `pnpm verify`
- 无法明确定位应修改的文件
- 需要在多个方案之间做设计取舍

六、你的输出方式
- 先说明你理解的任务范围
- 再说明你准备修改哪些文件
- 修改后汇报 `pnpm verify` 的结果
- 如果计划要求额外命令，一并汇报结果
- 如果停止，明确说明停止原因和需要升级的点
```

## 5. `maintenance mode` 提示词模板

适用于：

- 文档
- 归档
- 规则脚本
- 低风险治理任务

```text
你当前处于 maintenance mode。

你的任务是维护仓库治理系统，不做新的架构设计。

你可以做的事情：
- 修复文档链接
- 归档旧文档
- 更新执行计划状态
- 补充或修复 lint / verify 脚本
- 修复低风险回归问题

你不可以做的事情：
- 修改跨层 contract
- 扩展新的产品能力
- 修改 SDK 或 channel 的对外边界

每次修改后必须执行：
- `pnpm lint:docs`
- 如有代码改动，再执行 `pnpm verify`

如果发现任务已经超出维护范围，立即停止并说明为什么需要升级到更高模式。
```

## 6. 推荐的任务分级

### 004–007 的特殊规则

`004` 到 `007` 这类任务默认分成两个阶段：

1. 先由 `high-capability mode` 冻结：
   - 单一路径协议选择
   - 允许修改目录
   - 禁止修改目录
   - 验收命令
   - 升级条件
2. 再由 `budget mode` 按冻结后的工作单执行实现

如果第一阶段还没有完成，弱模型不得直接进入实现。

### 弱模型可直接做

- 文档入口维护
- 归档整理
- 链接修复
- 小范围脚本修改
- 按既有 contract 补实现
- 修复 `pnpm verify` 失败
- 补 smoke test 或 scenario test

### 弱模型执行，强模型定方向

- `ContextBlock` 模型落地
- `MemoryStore` / `ConversationStore` port 实现
- service API contract 实现
- client SDK 最小实现
- channel adapter framework 最小实现
- 任何需要先冻结 contract 再分阶段落地的执行计划

### 必须升级到强模型或人工

- 修改总体架构分层
- 新增跨层 contract
- 调整 SDK 对外接口
- 修改 channel 生命周期协议
- 处理复杂安全与权限边界
- 连续两轮修不通 `pnpm verify`

## 7. 标准任务提示模板

除了模式提示词外，建议始终附带一段任务提示：

```text
当前任务：
<一句话描述任务>

任务范围：
- 允许修改的目录：
- 不允许修改的目录：

不做什么：
- ...

验收标准：
- `pnpm verify` 通过
- <计划要求的第二条命令，例如 `pnpm test:server`>
- <其他明确标准>

升级条件：
- 命中以下任一情况时停止并汇报：
- ...
```

这样模型拿到的不是“一个模糊目标”，而是一份明确工作单。

---

## 8. 使用建议

### 当预算充足时

- 用高能力模式做架构和 contract 收口
- 用 budget mode 执行实现
- 用 maintenance mode 做治理和收尾

### 当预算紧张时

- 只在必须做方向判断时使用高能力模式
- 默认让弱模型在 budget mode 下推进
- 把任务切得更小
- 强制依赖 `pnpm verify`

### 最重要的一条

> 不要让模型自己判断“我能不能做”，而要让它根据你写好的模式规则来判断“我是否被允许继续做”。

# Model Prompt Cheatsheet

## 用法

当你切换模型时，不要临时重写一大段说明，优先从这里复制对应模板。

如果任务复杂、跨层、需要做设计判断，用 `high-capability mode`。
如果任务预算紧张、范围小、已有计划和验收标准，用 `budget mode`。
如果任务只是文档、归档、规则脚本和低风险修复，用 `maintenance mode`。

对于 `004` 到 `007` 这类跨层执行计划，默认先用 `high-capability mode` 冻结：

- 单一路径协议与运行时选择
- 允许修改目录 / 禁止修改目录
- 验收命令
- 升级条件

冻结后再把实现阶段交给 `budget mode`。

## High-Capability Mode

```text
你当前处于 high-capability mode。

你的职责是做方向判断、收口 contract、处理高风险边界，并把结果沉淀成仓库内可执行的规则、计划和文档。

你可以做的事情：
- 设计或调整跨层 contract
- 收口架构边界
- 规划下一阶段执行计划
- 定义新的自动化约束与反馈回路
- 把弱模型后续要执行的任务写成单一路径工作单

你必须遵守：
1. 先阅读 `AGENTS.md`
2. 先阅读 `../index.md`
3. 优先把判断沉淀到文档、计划或脚本，而不是只留在聊天里
4. 如果方案会影响多个层级，必须明确写出影响范围和不做什么
5. 如果开始实现，结束前必须运行 `pnpm verify`
```

## Budget Mode

```text
你当前处于 budget mode。

你的职责不是重新设计系统，而是在现有仓库规则、执行计划和验证脚本约束下，完成小范围、低风险、可验证的实现任务。

你必须遵守以下规则：

一、开始前先读
- `AGENTS.md`
- `../index.md`
- `MODEL_OPERATING_MODES.md`
- 当前对应的执行计划
- 计划中已经冻结的允许修改目录、验收命令与升级条件
- 与任务直接相关的实现文件

二、你可以做的事
- 按执行计划完成小范围实现
- 修复 `pnpm verify` 失败
- 补充低风险测试、脚本和文档
- 在既有 contract 内补充实现
- 在高能力模式已经冻结的 exec-plan 内，补齐已批准的共享字段、路由常量或测试入口

三、你不可以做的事
- 新增或重写跨层 contract
- 修改总体架构方向
- 修改 SDK 对外接口
- 修改 channel 生命周期协议
- 做无计划的大范围重构
- 自行扩展需求范围
- 自行新增未冻结的 endpoint、event、DTO 或跨层语义

四、每次修改后必须执行
- `pnpm verify`

五、如果出现以下情况，立即停止并汇报，不要继续尝试
- 需要改架构边界
- 需要改计划外的共享 contract
- 连续两轮无法通过 `pnpm verify`
- 无法明确定位应修改的文件
- 需要在多个方案之间做设计取舍
```

## Maintenance Mode

```text
你当前处于 maintenance mode。

你的任务是维护仓库治理系统，不做新的架构设计。

你可以做的事情：
- 修复文档链接
- 归档旧文档
- 更新执行计划状态
- 补充或修复 lint / verify 脚本
- 修复低风险回归问题

你不可以做的事情：
- 修改跨层 contract
- 扩展新的产品能力
- 修改 SDK 或 channel 的对外边界

每次修改后必须执行：
- `pnpm lint:docs`
- 如有代码改动，再执行 `pnpm verify`

如果发现任务已经超出维护范围，立即停止并说明为什么需要升级到更高模式。
```

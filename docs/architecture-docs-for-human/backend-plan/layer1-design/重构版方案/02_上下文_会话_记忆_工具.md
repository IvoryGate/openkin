# 02：上下文、会话、记忆与工具运行时

> 配套主文档：`00_第一层技术方案总览.md`  
> 主题：讲清 `ContextManager` 的工作方式、压缩扩展接口，以及工具注册与工具执行的边界  
> 工具运行时正式接口组补充见：`05_工具运行时接口组.md`  
> 本文档角色：可实现草案中的上下文装配、Session 资源与工具衔接说明

---

## 1. 为什么这份文档重要

第一层里最容易被看混的几个问题是：

1. `ContextManager` 到底什么时候开始工作？
2. 它和 `RunEngine` 的关系是平行关系，还是调用关系？
3. 它是否应该负责上下文压缩与摘要？
4. 工具到底什么时候注册，什么时候暴露给模型，什么时候真正执行？

这几个问题如果不讲清楚，后面的时序图和实现会越来越绕。

---

## 2. `ContextManager` 的定位

### 2.1 它是什么

`ContextManager` 是某个 `SessionRuntime` 的上下文工作区与预算控制器。

它的职责不是“保存一切”，而是：

> 维护当前要发给 LLM 的候选上下文，并在预算内输出一份高质量、结构合法的消息快照。

### 2.2 它不是什么

它不是：

- 会话历史数据库
- 长期记忆数据库
- 权限控制器
- 工具注册中心
- 任务调度器
- 多 Agent 协作器

### 2.3 它属于谁

它属于 `SessionRuntime`，而不是 `AgentDefinition`。

因为：

- 同一个 Agent 可以同时服务多个 Session
- 多个 Session 必须各自维护自己的上下文工作区

此外，Session 类型的扩展应采用**注册式扩展**，而不是开放任意字符串直接进入运行时。

---

## 3. `ContextManager` 的出现时机

### 3.1 对象存在时机

`ContextManager` 作为 `SessionRuntime` 的成员而存在。

也就是说，按推荐设计：

- 创建或激活 Session 时，会准备好 `SessionRuntime`
- `SessionRuntime` 中包含一个 `ContextManager`

因此，`ContextManager` 不是在 `run()` 过程中“突然创建出来的概念”，它在 Session 级别上就已经存在。

### 3.2 真正开始工作的时机

虽然它属于 SessionRuntime，但它真正开始参与消息装配，是在：

> 某次 `run()` 的第一个 step 中，首次调用 LLM 之前

这一步里，`RunEngine` 会把“本次运行需要的材料”装到 `ContextManager` 中。

---

## 4. `RunEngine` 与 `ContextManager` 的关系

### 4.1 不是纯平行关系

它们在模块图上是两个独立组件，但在运行时关系上不是对等协作，而是：

> `RunEngine` 驱动执行流程，`ContextManager` 被 `RunEngine` 调用来管理上下文工作区。

因此更准确的层级是：

```text
RunEngine -> SessionRuntime -> ContextManager
```

### 4.2 谁调用谁

调用关系是：

- `RunEngine` 调 `ContextManager`
- `ContextManager` 不主动推进流程

原因：

只有 `RunEngine` 知道：

- 当前是第几步
- 当前是否该调用 LLM
- 当前是否需要执行工具
- 当前是否该结束整个 `run()`

### 4.3 数据如何流动

数据流应当是 pull 模式：

1. `RunEngine` 准备基础材料
2. 把材料写入 `ContextManager`
3. `ContextManager` 组织、压缩、裁剪
4. `RunEngine` 再从中取出最终消息快照
5. 交给 LLM

也就是说，`ContextManager` 不主动“回填上下文”，而是被执行引擎读取。

---

## 5. 首次执行时，ContextManager 到底做什么

### 5.1 首次调用前必须已经装配好上下文

你的直觉是对的：

在首次调用 LLM 之前，确实应该已经把：

- `systemPrompt`
- 历史消息
- 当前用户输入
- 可能的首步记忆注入
- 额外上下文贡献

都拼好，再作为 react loop 的第一次输入。

### 5.2 这一步在第一层中的名字

在第一层设计里，这一步统一被抽象成：

- `build context`

它既包括第一次消息装配，也包括后续每一步的上下文重建。

### 5.3 首次输入前的真实动作

建议的首次装配流程如下：

1. `RunEngine` 读取 `AgentDefinition.systemPrompt`
2. 读取 `ConversationHistory` 恢复该 Session 历史
3. 追加本次用户输入
4. 调用 `onContextBuild` 收集长期记忆、RAG、额外上下文贡献
5. 将这些内容加载或追加到 `ContextManager`
6. 从 `ContextManager` 取出当前候选快照
7. 应用预算与压缩策略
8. 生成第一次发给 LLM 的 `messages[]`

---

## 6. `ContextManager` 的推荐调用序列

建议保留类似如下接口：

```typescript
contextManager.reset()
contextManager.loadBaseBlocks(baseBlocks)
contextManager.appendBlocks(dynamicBlocks)
const snapshot = contextManager.snapshot()
const finalMessages = await contextManager.ensureWithinBudget(policy)
```

这里的真实含义是：

- `RunEngine` 把材料放进去
- `ContextManager` 负责整理、分层和裁剪
- `RunEngine` 把最终结果拿出来交给 LLM

---

## 7. 为什么不能只装配一次上下文

### 7.1 因为 ReAct 是循环

ReAct 的本质是：

1. 调用 LLM
2. 看是否需要用工具
3. 如果用了工具，把结果回注上下文
4. 继续下一轮推理

所以不能只在第一次装配上下文，然后一直复用。

### 7.2 后续步骤会发生什么变化

当工具执行结束后，新的信息会进入上下文，例如：

- 网页抓取结果
- 数据库查询结果
- 代码执行输出
- 文件读取结果

因此下一轮 LLM 调用前，消息集合必须重新生成。

### 7.3 一句话总结

第一次是“首次装配”，后面每一次是“重建快照”，但两者在机制上都属于 `build context`。

---

## 8. `ConversationHistory`、`LongTermMemory`、`ContextManager` 的分工

### 8.1 `ConversationHistory`

这是 Session 级“档案室”。

负责：

- 读取该 Session 历史
- 写入本轮新增消息

不负责：

- 当前发送给 LLM 的拼装与裁剪

### 8.2 `LongTermMemory`

这是 Agent 级“经验库”。

负责：

- 跨 Session 保存经验、偏好、结论
- 在首步或指定时机注入相关记忆

不负责：

- 会话级消息历史还原

### 8.3 `ContextManager`

这是当前 `run()` 的上下文工作台。

负责：

- 维护当前候选上下文
- 控制 token 预算
- 组织压缩 / 外置 / 恢复策略
- 在工具调用后纳入新结果

不负责：

- 永久存储
- 记忆检索实现
- 具体的 LLM 摘要算法本身

---

## 9. 为什么 `ContextManager` 要支持“分层上下文”

如果未来要支持：

- 简单裁剪
- 分层摘要
- 将历史写入本地，仅保留引用
- 需要时按引用恢复原文
- 规则压缩与 LLM 压缩混合

那么上下文就不能再被看成一串平铺 `Message[]`，而应该先被抽象成“上下文块”。

建议引入：

```typescript
interface ContextBlock {
  id: string
  layer: 'system' | 'pinned' | 'recent' | 'history' | 'memory' | 'tool_result' | 'external_reference'
  messages: Message[]
  priority: number
  compressible: boolean
  restorable: boolean
  protectionLevel: 'immutable' | 'pinned' | 'compressible' | 'discardable'
  metadata?: Record<string, unknown>
}
```

这样做的好处是：

- 哪些块不能动，接口层就能表达
- 哪些块可摘要、可外置、可删除，策略层能精确处理
- “写出到本地，仅保留引用”会变成替换某个 block，而不是篡改整段消息数组

---

## 10. `ContextManager` 与压缩策略的职责分工

### 10.1 `ContextManager` 应负责什么

它应负责：

- 组织上下文块
- 做预算检查
- 决定是否进入压缩管线
- 调用压缩策略
- 输出最终消息集

### 10.2 它不应独自实现什么

它不应独自实现：

- 所有压缩算法
- 所有摘要逻辑
- 所有本地存储写入逻辑
- 所有恢复逻辑

### 10.3 推荐拆分

建议拆出以下接口：

- `CompressionPolicy`
- `CompressionStrategy`
- `ContextSummarizer`
- `ContextArtifactStore`
- `ContextRestorer`

这能保证未来无论压缩策略怎么演进，都不需要重写 `ContextManager` 主体。

---

## 11. 当需要用 LLM 做上下文总结时，应该如何工作

### 11.1 首期决策

首期压缩策略**只实现规则剪裁**，不直接落地 LLM 摘要。

也就是说：

- 当前文档保留 LLM 摘要接口
- 但第一阶段实现中，`ContextManager` 只调用 `TrimStrategy`
- `ContextSummarizer` 先作为预留接口，不强制提供实现

### 11.2 不建议让 `ContextManager` 直接耦合 provider

更合理的模式是：

- `ContextManager` 发现超预算
- 进入压缩管线
- 某个 `CompressionStrategy` 判断哪些块需要摘要
- 真正的摘要动作委托给 `ContextSummarizer`

### 11.3 推荐链路

1. `RunEngine` 请求最终上下文快照
2. `ContextManager` 检测超预算
3. 先进行规则裁剪
4. 仍超预算，则调用摘要策略
5. 摘要策略通过 `ContextSummarizer` 调 LLM 生成结构化摘要
6. 用摘要块替换原有长块
7. 再做预算检查
8. 输出最终消息集

### 11.4 为什么这样设计

因为这样可以做到：

- 让 `ContextManager` 保持为预算控制器和工作区协调器
- 让摘要能力可替换
- 让 LLM 压缩与规则压缩并存

---

## 12. 如何保证系统提示词不被稀释

这不是单纯“别删 systemPrompt”就能解决的问题，而是需要一套分层保护机制。

### 12.1 不可压缩层

`systemPrompt`、安全规则、Agent 身份定义、关键输出约束必须位于：

- 不可压缩层
- 不可外置层
- 不可摘要替代层

### 12.2 固定保留预算

第一层应支持分层预算概念，至少确保：

- 系统层有固定保留预算
- 当前任务目标与最新输入有固定保留预算
- 最近窗口有最低保留预算

### 12.3 历史摘要不能替代系统层语义

历史摘要只能补充背景，不能覆盖系统提示词本身。

正确顺序应该是：

```text
[System Prompt]
[Safety / Pinned Constraints]
[History Summary]
[Recent Window]
[Latest User Input]
```

而不是把系统规则混在摘要里。

---

## 13. 工具系统为什么也容易混

工具相关的问题，本质上有三层：

1. 工具定义
2. 工具注册
3. 工具使用 / 工具执行

很多方案之所以越写越绕，是因为把这三层混成了一个时机。

---

## 14. 工具注册时机

### 14.1 工具注册属于装配期

工具注册不是发生在某次 `run()` 的 step 里，而是发生在：

- Agent 初始化时
- Agent 能力装配时
- 或 Session 激活前的运行时准备阶段

在这个阶段，系统应先搞清楚：

- 这个 Agent 默认有哪些工具
- 这些工具来自哪里
- 是否允许暴露给模型

### 14.2 注册阶段的产物

注册完成后，至少应得到：

- 一个工具名到执行器的映射
- 一组能暴露给 LLM 的 `ToolSchema`
- 当前 Agent 生效的工具视图

---

## 15. 工具使用时机

### 15.1 暴露给 LLM

在每次调用 LLM 之前，`RunEngine` 从工具运行时拿到当前可用工具的 schema，并注入请求。

这个阶段做的是：

- “告诉模型你可以用哪些工具”

不是：

- “现场才开始注册工具”

### 15.2 真正执行工具

当 LLM 返回 `toolCalls` 后，`RunEngine` 按工具名去查对应执行器，然后真正执行。

### 15.3 工具结果回注

工具执行完后，结果会被转成 `tool message` 或等价结构，再 append 回 `ContextManager`。

然后下一步重新生成上下文。

---

## 16. 推荐的工具运行时分层

建议把工具相关能力拆成三层：

### 16.1 `ToolDefinition`

定义工具的静态信息：

- `name`
- `description`
- `inputSchema`

### 16.2 `ToolRegistry`

负责注册和查询：

- 注册工具
- 解决命名冲突
- 返回 Agent 当前可见工具列表

### 16.3 `ToolExecutor`

负责按名字执行工具：

- 根据 `ToolCall.name` 路由到具体实现
- 做统一错误处理
- 返回 `ToolResult`

---

## 17. 一个完整的首次 step 建议流程

把 `ContextManager` 和工具系统放在一起后，首次 step 应当是：

1. `RunEngine` 启动 `run()`
2. 读取 `SessionRuntime` 中的 `ContextManager`
3. 通过历史插件恢复历史
4. 组装 `systemPrompt + 历史 + 当前输入`
5. 调用 `onContextBuild`
6. 将额外上下文写入 `ContextManager`
7. 取快照并应用预算与压缩策略
8. 从工具运行时获取当前可用工具 schema
9. 调 LLM
10. 如果出现 `toolCalls`
11. 按名字执行工具
12. 将工具结果回注到 `ContextManager`
13. 进入下一步

---

## 18. 最终建议

第一层在实现时，应坚持以下规则：

1. `ContextManager` 属于 `SessionRuntime`
2. 首次调用 LLM 前必须先完成上下文首次装配
3. 后续每一步都应重新生成上下文快照
4. `ContextManager` 负责工作区与预算控制，但不负责控制流程
5. 压缩、摘要、外置、恢复必须通过独立策略接口扩展
6. `systemPrompt` 与关键约束必须具备硬保护
7. 工具注册发生在装配期，不发生在执行循环里
8. 工具 schema 暴露和工具执行是两个完全不同的时机
9. 工具结果必须重新回注上下文，成为后续推理的一部分
10. 首期压缩只落规则剪裁，实现保持最小；摘要与恢复接口先预留

---

## 19. 可实现草案：最小接口组

建议第一层先冻结下面这组最小接口，再进入实现：

```typescript
interface SessionRuntime {
  session: Session
  llm: LLMProvider
  contextManager: ContextManager
  toolRuntime: ToolRuntime
  hookRunner: HookRunner
  conversationStore?: ConversationStore
  memoryStore?: MemoryStore
}

interface ContextManager {
  buildSnapshot(state: RunState): Promise<Message[]>
  appendUserInput(input: AgentRunInput, state: RunState): Promise<void>
  applyToolResults(results: ToolResult[], state: RunState): Promise<void>
  commitAssistantMessage(message: Message, state: RunState): Promise<void>
}

interface ToolRuntime {
  getRuntimeView(state: RunState): Promise<ToolRuntimeView>
}
```

这三个接口分别对应：

- `SessionRuntime`：运行期资源聚合点
- `ContextManager`：上下文工作区与快照生成点
- `ToolRuntime`：当前步可见工具的提供者

## 20. 可实现草案：首次 `run()` 的最小落地顺序

首期编码时建议严格按下面顺序推进：

1. 先实现 `SessionRegistry -> SessionRuntime`
2. 再实现 `ContextManager.appendUserInput()`
3. 再实现 `ContextManager.buildSnapshot()`
4. 再实现 `ToolRuntime.getRuntimeView()`
5. 然后让 `ReActRunEngine` 打通第一次 LLM 调用
6. 最后补 `applyToolResults()` 和多 step 循环

这样做的原因是：

- 可以先验证一次 `LLM -> Final`
- 再验证一次 `LLM -> Tool -> LLM`
- 出问题时更容易定位到底是 Session、Context 还是 Tool 环节

## 21. 首期不建议在这里做的事情

为了保证第一层尽快形成可运行骨架，当前阶段不建议把下面这些能力直接压进 `ContextManager`：

- 向量检索
- 复杂自动摘要
- 多级召回排序
- 工具级权限系统
- 长历史外置落盘实现

这些都应先通过接口和 Hook 预留，不要让 `ContextManager` 在首期就变成“大总管”。

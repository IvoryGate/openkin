| # | 痛点 | 严重度 | 说明 |

| P3 | **群聊无法实时看到 Agent 在"思考"** | 高 | 当前群聊是并行等所有 Agent 回复后才一次性渲染，用户等待时无任何反馈 |

### Wave C：核心交互增强（5 个工单）

#### 155-C1 | @提及功能 — 输入 @弹出 Agent 选择器 + 强制指定回复

> **优先级**：P0 — 核心交互缺失
> **预估**：1.5d
> **前置**：153-B2

**目标**：
- 群聊输入框中输入 `@` 字符时，弹出 Agent 选择浮窗
- 选择后自动插入 `@AgentName` 到输入框
- 消息发送后，被 @ 的 Agent **必须回复**（跳过 Self-Judge 的 [SKIP] 判断）
- 消息气泡中 `@AgentName` 高亮显示
- 未被 @ 的 Agent 仍走 Self-Judge 路由

**实现细节**：

1. **@弹出选择器**：
   - 监听 `channel-composer-input` 的 `input` 事件
   - 当检测到光标前一个字符是 `@` 时，在输入框上方弹出浮窗
   - 浮窗列出当前群聊的所有 Agent（头像 + 名称 + 在线状态）
   - 键盘上下键选择，Enter/点击确认
   - 输入 `@x` 时过滤名称包含 "x" 的 Agent
   - 按 Esc 或点击外部关闭浮窗

2. **消息文本解析**：
   - 发送前用正则提取所有 `@AgentName`，匹配到对应 agentId
   - 将匹配结果附加到消息元数据中

3. **群聊路由修改**：
   - 被 @ 的 Agent：`enrichedText` 中移除 [回复规则] 里的 SKIP 指令，改为"你被 @ 了，必须回复"
   - 未被 @ 的 Agent：保持原有 Self-Judge 逻辑

4. **气泡渲染**：
   - `@AgentName` 用特殊样式渲染（蓝色/品牌色高亮，加粗）

5. **私聊中**：
   - 私聊只有一个 Agent，@ 功能无意义，但可以保留语法（自动忽略）

**涉及文件**：
- `apps/desktop/renderer/app.js` — @检测、浮窗逻辑、路由修改、气泡渲染
- `apps/desktop/renderer/styles.css` — @浮窗样式、高亮样式
- `apps/desktop/renderer/index.html` — @浮窗容器

#### 155-C2 | 群聊流式回复可见 — 每个 Agent 实时打字效果

> **优先级**：P0 — 体验断裂
> **预估**：2d
> **前置**：153-B2

**目标**：
- 群聊发消息后，每个 Agent 的回复过程**实时流式可见**
- 每个 Agent 有独立的"正在输入…"指示器
- Agent 回复逐字出现，类似私聊的打字效果
- 不同 Agent 的流式回复互不干扰，可并行显示

**当前问题**：
当前 `sendChannelGroupMessage` 用 `Promise.allSettled` 等待所有 Agent 的 Run 完成，然后一次性把所有回复加入消息列表。用户在等待期间只看到自己的消息，没有任何反馈，体验很差。

**实现细节**：

1. **改为逐个 Agent 流式**：
   - 不再等所有 Agent 完成后一次性渲染
   - 每个 Agent 开始 Run 后，立即在消息列表中添加一个"正在输入"占位气泡
   - 随着流式 delta 到达，实时更新该 Agent 的气泡内容
   - Agent 完成（或 [SKIP]）后，移除占位或保留最终内容

2. **流式状态追踪**：
   ```
   channelGroupStreaming = {
     [convId]: {
       [agentId]: { traceId, buffer, status: "streaming" | "done" | "skipped" }
     }
   }
   ```

3. **UI 展示**：
   - 每个"正在输入"的 Agent 显示：头像 + 名字 + `⋯` 动画
   - 有 delta 内容后，`⋯` 替换为实时文字（逐字出现）
   - Agent 完成/跳过后，最终渲染为正常消息气泡或直接移除

4. **取消机制**：
   - 用户可以在 Agent 回复过程中发送新消息
   - 正在进行的流式回复不受影响

**涉及文件**：
- `apps/desktop/renderer/app.js` — 重构 `sendChannelGroupMessage` 为流式模式
- `apps/desktop/renderer/styles.css` — 多 Agent 并行输入指示器样式

#### 155-C3 | 消息操作菜单（复制/回复/重发/删除）

> **优先级**：P1 — 基础 IM 体验
> **预估**：1.5d
> **前置**：无

**目标**：
- 消息气泡 hover 时出现操作按钮（或右键弹出菜单）
- 支持：复制内容、回复引用、重新生成（Agent 消息）、删除

**实现细节**：

1. **Hover 操作栏**：
   - 用户消息 hover：显示「复制 | 回复 | 删除」
   - Agent 消息 hover：显示「复制 | 回复 | 重新生成 | 删除」
   - 操作栏浮在气泡右上角，半透明背景

2. **复制**：
   - 点击后复制消息纯文本到剪贴板
   - 使用 `navigator.clipboard.writeText()`

3. **回复引用**：
   - 点击后输入框自动出现引用预览条
   - 引用条显示"回复 [AgentName]：消息前20字…"
   - 发送时在消息元数据中附加 `replyTo: { msgId, senderName, content }`
   - 气泡上方显示引用小条，点击可跳转到原消息

4. **重新生成**：
   - 仅 Agent 消息可用
   - 用同一条用户消息重新发起 Run
   - 新回复替换旧回复（或追加，待定）

5. **删除**：
   - 从本地消息存储中移除
   - 二次确认（仅 Agent 消息需要，用户消息直接删）

6. **右键菜单**（可选）：
   - 与 hover 操作栏功能相同，但以右键菜单形式呈现

**涉及文件**：
- `apps/desktop/renderer/app.js` — 操作栏渲染、各操作逻辑、回复引用数据模型
- `apps/desktop/renderer/styles.css` — 操作栏样式、引用条样式
- `apps/desktop/renderer/index.html` — 可能需要右键菜单容器

#### 155-C4 | 消息 Markdown 渲染增强

> **优先级**：P1 — Agent 回复质量差
> **预估**：1d
> **前置**：无

**目标**：
- 频道消息气泡中的 Agent 回复支持 Markdown 渲染
- 包括：代码块（语法高亮）、加粗/斜体、列表、链接、表格
- 复用聊天 Tab 已有的 Markdown 渲染能力

**当前问题**：
频道消息当前用 `escapeHtml(msg.content)` 渲染，Agent 回复中的代码块、列表等全部显示为纯文本，可读性极差。

**实现细节**：

1. **渲染策略**：
   - Agent 消息：使用 Markdown 渲染器（复用 `renderMarkdown` 函数或 marked.js）
   - 用户消息：保持纯文本（用户消息通常不含 Markdown）

2. **代码块**：
   - 语法高亮（如已集成 highlight.js）
   - 代码块右上角"复制"按钮
   - 与聊天 Tab 代码块样式一致

3. **链接**：
   - 自动识别 URL 并转为可点击链接
   - `target="_blank"` 在新窗口打开

4. **XSS 防护**：
   - Markdown 渲染需做 sanitize，防止 Agent 回复中注入恶意 HTML/JS

**涉及文件**：
- `apps/desktop/renderer/app.js` — 引入 Markdown 渲染，修改 `renderChannelMessages`
- `apps/desktop/renderer/styles.css` — 气泡内 Markdown 样式

### Wave E：高级功能与完整 IM 体验（4 个工单）

#### 155-E1 | 频道消息历史持久化 — 服务端加载 + 本地缓存

> **优先级**：P0 — 数据可靠性
> **预估**：2d
> **前置**：153-A4

**目标**：
- 频道私聊消息从服务端 Session 加载历史（不再纯依赖 localStorage）
- 消息加载支持分页（滚动到顶部加载更早消息）
- 本地缓存作为快速加载层，服务端数据作为权威数据源
- 群聊消息仍使用本地存储（因为群聊是客户端概念，后端不理解"群"）

**实现细节**：

1. **私聊消息加载策略**：
   - 选中联系人时，先从 localStorage 快速渲染本地缓存
   - 然后异步从 `GET /v1/sessions/:id/messages` 加载服务端消息
   - 合并去重（按 timestamp + content 匹配）
   - 服务端数据覆盖本地缓存

2. **分页加载**：
   - 初始加载最近 50 条
   - 滚动到顶部时，用 `before` 游标加载更早的 50 条
   - 加载时顶部显示加载指示器

3. **消息同步**：
   - 新消息发送后同时写入本地缓存和发送到服务端
   - 服务端消息作为最终真实来源

4. **群聊消息**：
   - 群聊消息仍存 localStorage（后端没有"群"概念）
   - 每个群聊的每个 Agent 有独立 Session，可选择性加载各 Agent Session 的历史
   - 首期不做群聊消息的服务端加载（复杂度高，收益低）

**涉及文件**：
- `apps/desktop/renderer/app.js` — 消息加载重构、分页逻辑、合并去重
- `apps/desktop/renderer/styles.css` — 加载指示器样式

#### 155-E3 | 会话导出 + 对话统计

> **优先级**：P2 — 实用功能
> **预估**：1d
> **前置**：155-C4

**目标**：
- 支持将单个会话导出为 Markdown 文件
- 支持导出为 JSON（完整消息元数据）
- 提供简单的对话统计（消息数、Agent 参与度等）

**实现细节**：

   - 聊天标题栏右侧 ⋯ 菜单 → "导出对话"

   - 完整消息列表，包含所有元数据（id、sender、timestamp、avatarColor 等）

4. **对话统计**：
   - 会话信息面板底部显示：
     - 总消息数

#### 155-E4 | Router Agent 模式 — 智能群消息分发

> **优先级**：P3 — 高级编排
> **预估**：2d
> **前置**：155-C1、155-C2

**目标**：
- 群聊支持指定一个 Agent 作为"Router Agent"
- 用户发消息后，Router Agent 先分析消息意图，决定应该由哪些 Agent 回复
- 只有被 Router Agent 选中的 Agent 才会收到消息并回复
- 与 Self-Judge 模式可切换

**实现细节**：

1. **群聊路由模式**：
   - `routingMode: "self-judge" | "router"`
   - 默认 `self-judge`（现有逻辑）
   - `router` 模式下需要指定 `routerAgentId`

2. **Router 工作流**：
   ```
   用户消息 → Router Agent（分析意图，输出 JSON：应回复的 Agent ID 列表）
           → 只有被选中的 Agent 收到消息并回复
           → 其他 Agent 不参与
   ```

3. **Router Agent 的 Prompt**：
   - 注入群名、成员列表、成员角色描述
   - 要求输出格式：`{"replyAgents": ["agent-id-1", "agent-id-2"], "reason": "..."}`
   - 如果不需要任何人回复，返回空列表

4. **群设置中切换路由模式**：
   - 在群聊设置面板增加"路由模式"选择
   - 可选：自由讨论（Self-Judge）/ 智能分发（Router）
   - Router 模式下选择 Router Agent

5. **性能考虑**：
   - Router 模式增加一次 LLM 调用延迟（路由判断）
   - 但减少了不必要的并行 Run，总体可能更快

**涉及文件**：
- `apps/desktop/renderer/app.js` — Router 路由逻辑、群设置增加路由模式
- `apps/desktop/renderer/styles.css` — 路由模式选择器样式

## 七、风险与应对

| 风险 | 应对 |
|------|------|
| 群聊流式回复需要管理多个并行 SSE 连接 | 每个 Agent 的 stream 独立追踪，互不干扰 |
| Markdown 渲染可能影响性能（长消息） | 使用 requestAnimationFrame 分批渲染 |
| @提及的 Agent 名称匹配可能有歧义 | 用 agentId 而非名称匹配，显示名仅用于展示 |
| Router Agent 模式增加延迟 | Router 用轻量模型/Fast Agent，减少判断时间 |
| 消息持久化重构可能引入数据丢失 | 保留 localStorage 作为 fallback，服务端数据覆盖而非替换 |
| 聊天搜索性能（大量消息时） | 首期限定搜当前会话，全局搜索做异步分批 |

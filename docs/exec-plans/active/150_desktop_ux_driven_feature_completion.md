#### P0 — 核心交互断裂（用户会卡住或收到误导反馈）

| # | 缺口 | 现状 | 影响 |
|---|------|------|------|
| P0-1 | **附件上传无实际功能** | `uploadAttachmentEl` 只切换 `hasAttachment` 标志，发送时塞入 `desktop://attachment-placeholder` 占位 | 用户点击"上传附件"或"上传图片"后没有任何文件选择器，发送后后端收到伪造数据 |
| P0-2 | **联网检索开关无效** | `networkEnabled` 只是本地状态，代码中写了 `run_hint(network=off)，协议暂不支持该策略字段，已按默认策略运行` | 用户以为关闭了联网，实际后端仍然联网 |
| P0-3 | **完全控制开关无效** | 同上，`fullControlEnabled` 仅本地 UI 状态 | 用户以为开启了高危权限，实际无影响 |
| P0-4 | **会话无法删除** | 后端已有 `DELETE /v1/sessions/:id`，但客户端无入口 | 会话列表只增不减，无法清理废弃会话 |

| P0-6 | **消息无分页加载** | `getSessionMessages` 只取 `?limit=100`，没有 `before` 游标翻页 | 长对话超过 100 条后丢失早期消息，无法回溯 |
| P0-7 | **模型选择与 Agent 脱节** | `MODEL_TO_AGENT_ID` 三个模型映射值全是空字符串 | 选择"theworld-Fast"或"theworld-Pro"对 run 无任何影响，因为 agentId 永远为空 |

## 二、用户旅程缺口分析

### 旅程 1：新用户首次使用

```
打开 App → 看到空态 → 点击"新建" → 选 Agent → 开始对话 → 得到回复
```

**缺口**：
- 新建会话时 Agent Picker 依赖已初始化的预设，但预设初始化需要后端在线；如果后端离线，Picker 显示空列表且无明确提示
- Hero 区文本 "我是 theworld" 是硬编码，不反映所选 Agent 身份
- 首次使用无引导 / onboarding

### 旅程 2：日常多会话切换

```
查看会话列表 → 找到目标会话 → 切换 → 继续对话
```

**缺口**：
- 搜索缺失，会话多了无法快速定位
- 会话标题都是 Agent 名，同 Agent 多会话无法区分（无重命名）
- 列表只显示最近 8 条，需要点"查看全部"进入另一个全屏视图，来回切换成本高

### 旅程 3：长对话深度使用

```
查看历史 → 滚动到早期消息 → 回复 → 查看 Agent 推理过程 → 做决策
```

**缺口**：
- 超过 100 条消息的历史不可见（无分页）
- 推理过程/工具调用展示只依赖 localStorage 归档，换设备/清缓存后丢失
- 消息无操作菜单，无法对单条消息做重发/删除/复制

### 旅程 4：文件与多媒体交互

```
上传附件 → 发送 → Agent 读取文件 → 回复
```

**缺口**：
- 附件和图片上传按钮完全是空壳，无法选择文件
- 联网检索开关不生效，用户无法控制 Agent 是否联网

### 旅程 5：定时任务与自动化

```
创建定时任务 → 查看 Task 列表 → 触发/禁用 → 查看运行历史
```

**缺口**：
- 完全没有 Task 管理界面（后端已具备完整 CRUD API）

## 三、开发方向与工单拆分

基于以上分析，按"用户体验严重程度 × 实现复杂度"排序，分 5 个波次推进。

### Wave 1：核心交互修复（消除 P0 断裂点）

> **目标**：让现有 UI 中每一个可点击的元素都有真实功能，消除"点击无反应"或"假装有效"的体验。

#### WO-1A：会话生命周期补全

| 子项 | 内容 | 预估 |
|------|------|------|
| 会话删除 | 右键 / 长按 / 下拉菜单添加"删除"，调用 `DELETE /v1/sessions/:id` | 0.5d |
| 会话重命名 | 双击会话标题可编辑，调用 `PATCH /v1/sessions/:id` 更新 `displayName` | 0.5d |
| 消息分页加载 | 滚动到顶部时加载更早消息（`before` 游标），支持无限滚动 | 1d |
| 查询单个会话 | bridge 补全 `getSession`，会话切换时刷新元数据 | 0.5d |

#### WO-1B：Run 策略与附件真实化

| 子项 | 内容 | 预估 |
|------|------|------|
| 联网检索 → Agent 级别控制 | 将 `networkEnabled` 映射到选择带/不带联网工具的 Agent，或传递 `executionMode` | 1d |
| 完全控制 → 风险等级映射 | 将 `fullControlEnabled` 映射到 approval 策略（高风险操作自动弹出审批而非静默允许） | 1d |
| 文件上传 | 点击"上传附件"触发 Electron `dialog.showOpenDialog`，读取文件后通过 preload 桥传递给 renderer，创建 `file` 类型的 attachment | 1.5d |
| 图片上传 | 同上，选择图片文件后转为 `image` attachment（支持 base64 URL 或临时路径） | 0.5d |
| 模型-Agent 映射修正 | 填充 `MODEL_TO_AGENT_ID`，或在 Agent Picker 中直接选择带指定模型的 Agent | 0.5d |

### Wave 2：管理能力补全（消除结构性缺口）

## 四、工单执行顺序与依赖

```
Wave 1 ─── Wave 2 ─── Wave 3 ─── Wave 4 ─── Wave 5
  │            │          │          │          │
  ├─ WO-1A     ├─ WO-2A   ├─ WO-3A   ├─ WO-4    ├─ WO-5
  └─ WO-1B     └─ WO-2B   └─ WO-3B   └─         └─
```

- **Wave 1** 无外部依赖，可立即启动
- **Wave 2** 中 WO-2A 的 Task bridge 依赖 shared-contracts 中已有的 Task 路由常量
- **Wave 3** 的上下文真实用量依赖 `GET /v1/runs/:traceId/context` 后端已部署
- **Wave 4** 依赖 Wave 1 的消息分页基础设施
- **Wave 5** 依赖 Wave 2 的 Task 管理、Wave 3 的 Tab 绑定

## 五、验收标准

每个 Wave 完成时：

1. **零空壳**：所有可点击的 UI 元素都有实际功能或明确标注"即将推出"
2. **零伪造**：所有数据来自真实后端 API，不存在 placeholder / 硬编码假数据
3. **pnpm verify 通过**：`pnpm verify` 和 `pnpm --filter @theworld/desktop check` 均无错误
4. **手动验收**：按用户旅程走一遍完整流程，无卡顿点

## 六、风险与约束

| 风险 | 应对 |
|------|------|
| 后端某些 API 可能未完全部署 | Wave 启动前先验证对应 API 可用性 |
| 联网/完全控制开关需要后端策略字段支持 | 若后端暂不支持，先在 UI 层面说明"该功能需后端升级" |
| 附件上传涉及文件系统安全 | 严格走 Electron 安全路径：`dialog` → preload → renderer，不暴露完整路径 |
| Task 管理可能需要新的 UI 空间 | 复用"工作台"Tab，不新增独立窗口 |

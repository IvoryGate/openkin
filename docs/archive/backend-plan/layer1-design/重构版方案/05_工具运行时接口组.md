# 05：工具运行时接口组

> 配套主文档：`00_第一层技术方案总览.md`  
> 主题：把第一层工具运行时从“概念边界”收口成可实现的正式接口组，确保第二层 `MCP / Skill / 内置工具` 能稳定接入  
> 本文档角色：可实现草案中的工具 contract 与默认落地方式

---

## 1. 为什么需要这份文档

当前第一层已经明确了：

- 工具定义
- 工具注册
- 工具使用

是三个不同阶段。

但如果只停留在概念层，第二层在落 `MCP / Skill / Tool Registry` 时仍然会反向侵入第一层：

- 执行引擎可能被迫感知工具来源
- Session 细节可能泄露给工具层
- 上层权限逻辑可能直接塞进工具执行器

因此第一层必须正式定义一组工具运行时接口。

---

## 2. 工具运行时要解决什么问题

第一层的工具运行时不负责“提供所有工具”，它负责解决这些基础问题：

1. 工具长什么样
2. 工具何时注册
3. 某个 Agent 当前能看到哪些工具
4. 某次 `run()` 中实际暴露给 LLM 的工具集合是什么
5. LLM 返回 `toolCall` 后，如何按名字路由执行
6. 工具执行时能拿到哪些运行时上下文

---

## 3. 工具运行时的四层模型

建议把第一层工具能力拆成四层：

### 3.1 `ToolDefinition`

回答：

> 这个工具是什么？

包含静态信息：

- `name`
- `description`
- `inputSchema`
- 输出约定

### 3.2 `ToolProvider`

回答：

> 这些工具来自哪里？

工具来源可以是：

- 第一层内置工具
- 第二层 Skill
- 第二层 MCP
- 第三层或业务侧注入工具

### 3.3 `ToolRuntimeView`

回答：

> 在当前 Agent / 当前 run 里，哪些工具真正可见？

它是一个经过筛选后的可见工具视图。

### 3.4 `ToolExecutor`

回答：

> 当前 LLM 选中了一个工具之后，如何真正执行？

---

## 4. 推荐接口组

### 4.1 `ToolDefinition`

```typescript
interface ToolDefinition {
  name: string
  description: string
  inputSchema: JsonSchema
  outputMode?: 'text' | 'json' | 'binary_ref'
  metadata?: Record<string, unknown>
}
```

### 4.2 `ToolProvider`

```typescript
interface ToolProvider {
  id: string
  sourceType: 'builtin' | 'skill' | 'mcp' | 'custom'
  listTools(): Promise<ToolDefinition[]>
  resolveExecutor(toolName: string): Promise<ToolExecutor | null>
}
```

说明：

- 第一层不关心它是 Skill 还是 MCP 实现
- 第一层只关心它是否能列出工具，并能解析出执行器

### 4.3 `ToolAccessPolicy`

```typescript
interface ToolAccessPolicy {
  filterVisibleTools(args: {
    agent: AgentDefinition
    session: Session
    toolDefinitions: ToolDefinition[]
    metadata?: Record<string, unknown>
  }): Promise<ToolDefinition[]>
}
```

说明：

- 这是“运行前筛选”
- 它决定某个 Agent 当前能看到哪些工具
- 不负责执行时的权限拦截

### 4.4 `ToolRuntimeView`

```typescript
interface ToolRuntimeView {
  toolDefinitions: ToolDefinition[]
  getToolSchemaList(): ToolDefinition[]
  resolve(toolName: string): Promise<ToolExecutor | null>
}
```

说明：

- 这是某次运行或某个 Agent 的可见工具快照
- `RunEngine` 在每次调用 LLM 前只依赖这个视图

### 4.5 `ToolExecutionContext`

```typescript
interface ToolExecutionContext {
  traceId: string
  sessionId: string
  agentId: string
  stepIndex: number
  requester?: {
    type: 'user' | 'agent' | 'system'
    id: string
  }
  metadata?: Record<string, unknown>
}
```

说明：

- 工具拿到的是“执行上下文”
- 不是直接拿 `SessionRuntime`
- 更不能直接操作 `ContextManager`

### 4.6 `ToolExecutor`

```typescript
interface ToolExecutor {
  execute(input: unknown, ctx: ToolExecutionContext): Promise<ToolResult>
}
```

---

## 5. 第一层中的工具生命周期

### 5.1 装配期

装配期完成：

1. 注册 `ToolProvider`
2. 聚合工具定义
3. 通过 `ToolAccessPolicy` 生成某 Agent 的可见工具集合

### 5.2 运行前

运行前完成：

1. 基于当前 Agent 和运行条件拿到 `ToolRuntimeView`
2. 将 `toolDefinitions` 转成 LLM 可识别 schema
3. 作为 `tools` 注入模型请求

### 5.3 运行时

运行时完成：

1. LLM 返回 `toolCalls`
2. `RunEngine` 从 `ToolRuntimeView` 解析执行器
3. 构造 `ToolExecutionContext`
4. 执行工具
5. 将 `ToolResult` 回注到上下文

---

## 6. 为什么要有 `ToolRuntimeView`

如果没有这一层，执行引擎就会被迫：

- 直接遍历所有 ToolProvider
- 自己做工具来源判断
- 自己做运行时筛选

这会带来两个问题：

1. 第二层工具生态实现会反向污染第一层执行引擎
2. Agent 当前“可见工具集”无法被清晰建模

所以推荐：

> 工具来源属于 `ToolProvider`，当前可见集合属于 `ToolRuntimeView`，执行引擎只依赖 `ToolRuntimeView`。

---

## 7. 为什么要有 `ToolExecutionContext`

如果工具执行时直接拿到：

- `SessionRuntime`
- `ContextManager`
- 整个 `RunState`

那工具层很容易开始直接篡改第一层运行时。

这会导致：

- 工具污染上下文边界
- Session 细节泄露
- 调试和审计困难

因此工具执行时只应拿到一份受限、稳定、可审计的 `ToolExecutionContext`。

---

## 8. 权限应该放在哪两层

工具权限至少分两层：

### 8.1 运行前可见性控制

由 `ToolAccessPolicy` 决定：

- 这个 Agent 当前看不看得到某工具

### 8.2 运行时拦截

由 Hook 或守卫决定：

- 模型虽然看到了工具，但这次调用是否允许执行

例如：

- `ToolAccessPolicy`：普通 Agent 不显示危险工具
- `onBeforeToolCall`：即便看到了，当前用户也可能无权执行

---

## 9. 与第二层的边界

### 第一层负责

- 工具模型
- 工具运行时
- 工具执行上下文
- 工具执行结果 contract

### 第二层负责

- Skill 组织工具
- MCP 接入远端工具
- 工具注册中心的具体上层来源管理

换句话说：

> 第一层定义“工具如何被运行”，第二层定义“工具从哪里来、如何被组织”。

---

## 10. 推荐的首期实现范围

为了避免首期过重，建议先落：

1. `ToolDefinition`
2. `ToolProvider`
3. `ToolRuntimeView`
4. `ToolExecutionContext`
5. `ToolExecutor`
6. 一个最小 `ToolAccessPolicy`

首期可以暂不做：

- 复杂多 provider 冲突解决
- Tool 级缓存
- Tool 级速率限制器
- provider 健康探测

但接口应先留好。

---

## 11. 最终建议

建议把第一层工具能力正式冻结为：

- **来源抽象：`ToolProvider`**
- **可见视图：`ToolRuntimeView`**
- **执行上下文：`ToolExecutionContext`**
- **执行器接口：`ToolExecutor`**
- **可见性策略：`ToolAccessPolicy`**

这样第二层无论接 Skill、MCP 还是更多工具来源，都不会反向打破第一层执行内核。

---

## 12. 可实现草案：推荐文件划分

建议工具运行时首期先拆成以下文件：

- `tool-runtime/types.ts`：`ToolDefinition`、`ToolResult`
- `tool-runtime/provider.ts`：`ToolProvider`
- `tool-runtime/access-policy.ts`：`ToolAccessPolicy`
- `tool-runtime/runtime-view.ts`：`ToolRuntimeView`
- `tool-runtime/executor.ts`：`ToolExecutor`、`ToolExecutionContext`

这样后续第二层即使接入更多来源，也只是在 provider 层扩展，不会反向污染执行引擎。

## 13. 可实现草案：默认最小实现链路

首期建议默认只支持：

1. 一个内置 `StaticToolProvider`
2. 一个最小 `DefaultToolAccessPolicy`
3. 一个运行时聚合器 `DefaultToolRuntime`

默认链路可以写成：

```typescript
class DefaultToolRuntime implements ToolRuntime {
  constructor(
    private readonly providers: ToolProvider[],
    private readonly accessPolicy: ToolAccessPolicy,
  ) {}

  async getRuntimeView(state: RunState): Promise<ToolRuntimeView> {
    const toolDefinitions = await this.collectDefinitions()
    const visibleTools = await this.accessPolicy.filterVisibleTools({
      agent: state.agent,
      session: state.session,
      toolDefinitions,
    })
    return createRuntimeView(visibleTools, this.providers)
  }
}
```

这样第一层就已经能支撑：

- 内置工具
- 上层注入工具
- 未来 Skill / MCP provider

## 14. 当前故意不定死的点

为了不提前锁死上层能力，下面这些点本轮仍保持开放：

1. `ToolResult` 是否内置 `structuredOutput`
2. 多个 provider 出现同名工具时的冲突策略
3. provider 是否需要心跳、健康度、可用性探测

也就是说，第一层先冻结“运行时边界”，而不是把所有工具生态问题一次讲死。

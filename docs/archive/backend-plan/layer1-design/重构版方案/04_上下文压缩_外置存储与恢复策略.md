# 04：上下文压缩、外置存储与恢复策略

> 配套主文档：`00_第一层技术方案总览.md`  
> 主题：为未来多种上下文压缩策略、历史外置存储、引用保留与按需恢复预留统一扩展接口  
> 本文档角色：可实现草案中的压缩管线与扩展接口说明

---

## 1. 为什么需要单独一份文档

一开始的 `ContextManager` 很容易只被理解成：

- 存消息
- 做 token 裁剪

但如果未来系统会出现：

- 多种上下文压缩策略
- LLM 摘要
- 长历史写入本地，只保留引用
- 需要时再恢复原文
- 外置片段按相关性召回

那么上下文管理就不能再停留在“消息数组裁剪”这个层面。

因此需要一套专门的扩展设计，把：

- 工作区管理
- 预算控制
- 压缩策略
- 外置存储
- 按需恢复

彻底拆开。

---

## 2. 核心设计原则

### 2.1 `ContextManager` 负责编排，不负责实现所有算法

`ContextManager` 应负责：

- 组织候选上下文
- 计算预算
- 触发压缩管线
- 产出最终消息快照

它不应独自承担：

- 所有压缩算法
- 所有 LLM 摘要逻辑
- 所有存储逻辑
- 所有恢复逻辑

### 2.2 上下文必须先分层，再谈压缩

未来压缩策略之所以容易失控，是因为很多实现直接拿平铺 `Message[]` 做处理。

正确做法应是：

> 先把上下文建模成分层的 `ContextBlock[]`，再让不同策略作用于这些 block。

### 2.3 所有高级压缩能力都应通过接口扩展

包括：

- 规则裁剪
- 结构化摘要
- 外置存储
- 引用替换
- 按需恢复
- 相关性召回

都不应写死在 `ContextManager` 本体中。

---

## 3. 上下文分层模型

### 3.1 为什么不能只用 `Message[]`

因为未来你要处理的上下文并不都是同等重要的。

有些内容：

- 绝不能删
- 可以摘要但不能删
- 可以外置但需要引用
- 可以直接丢弃

如果没有分层模型，策略就无法精确表达这些差异。

### 3.2 推荐的 `ContextBlock`

```typescript
type ContextLayer =
  | 'system'
  | 'pinned'
  | 'recent'
  | 'history'
  | 'memory'
  | 'tool_result'
  | 'external_reference'

type ProtectionLevel =
  | 'immutable'
  | 'pinned'
  | 'compressible'
  | 'discardable'

interface ContextBlock {
  id: string
  layer: ContextLayer
  messages: Message[]
  priority: number
  tokenEstimate: number
  protectionLevel: ProtectionLevel
  restorable: boolean
  metadata?: Record<string, unknown>
}
```

### 3.3 推荐的层次

#### `system`

包括：

- `systemPrompt`
- 安全规则
- Agent 身份定义
- 输出格式约束

要求：

- `immutable`

#### `pinned`

包括：

- 当前任务目标
- 当前用户最新输入
- 本轮关键约束

要求：

- 不能删除
- 尽量不摘要

#### `recent`

包括：

- 最近若干轮用户/助手对话
- 最近一到两步关键工具结果

要求：

- 优先保留原文

#### `history`

包括：

- 更早的对话历史
- 已被消费的老观察结果

要求：

- 允许摘要
- 允许外置

#### `memory`

包括：

- 长期记忆注入块

要求：

- 可按策略裁剪

#### `tool_result`

包括：

- 本次工具执行产生的结果块

要求：

- 长结果优先裁剪或外置

#### `external_reference`

包括：

- 已外置内容的引用块

要求：

- 自身通常较短
- 可按需恢复原文

---

## 4. 为什么系统提示词不会被“稀释”

这不是靠“尽量别删 systemPrompt”解决的，而是靠接口级保护。

### 4.1 系统层必须是 `immutable`

任何压缩、外置、恢复策略都不能处理 `immutable` 块。

### 4.2 必须支持分层预算

推荐预算观念不是只看“总 token 数”，而是同时看：

- 系统层保底预算
- pinned 层保底预算
- recent 层最低保留预算
- history 层动态压缩预算

### 4.3 历史摘要不能替代系统层语义

历史摘要只能作为背景补充，而不是把系统层“改写成摘要”。

因此顺序应当类似：

```text
[System Prompt]
[Pinned Constraints]
[History Summary]
[Recent Window]
[Latest Input]
```

而不是把系统规则混入历史摘要中。

---

## 5. 压缩策略不应该只有一种

未来上下文压缩很可能会同时存在多种策略：

- 规则裁剪
- 固定模板摘要
- LLM 摘要
- 外置存储
- 相关性恢复

因此第一层不应设计成：

- 一个 `compress()` 大函数

而应设计成：

- 多策略管线

---

## 6. `CompressionPolicy`：决定“该怎么压”

### 6.1 它负责什么

`CompressionPolicy` 负责决定：

- 当前上下文是否需要压缩
- 优先使用哪类策略
- 哪些层可动，哪些层不可动
- 目标压缩幅度是多少

### 6.2 推荐接口

```typescript
interface CompressionPolicy {
  name: string
  buildPlan(input: {
    blocks: ContextBlock[]
    tokenBudget: number
    currentUsage: number
    stepIndex: number
    isFirstStep: boolean
  }): CompressionPlan
}

interface CompressionPlan {
  hardPinnedLayers: ContextLayer[]
  preferredStrategyOrder: string[]
  targetReductionTokens: number
  allowExternalization: boolean
  allowLLMSummary: boolean
}
```

### 6.3 为什么需要单独的 Policy

因为未来很可能有多套不同偏好的策略，例如：

- `ConservativePolicy`：尽量不摘要，只做裁剪
- `BalancedPolicy`：裁剪 + 摘要混合
- `AggressivePolicy`：允许外置和大规模摘要
- `LowLatencyPolicy`：禁止 LLM 摘要

---

## 7. `CompressionStrategy`：负责“具体怎么压”

### 7.1 它负责什么

每个 `CompressionStrategy` 只负责一种具体处理手段。

例如：

- `TrimStrategy`
- `SummarizeStrategy`
- `ExternalizeStrategy`
- `RestoreStrategy`
- `ReorderStrategy`

### 7.2 推荐接口

```typescript
interface CompressionContext {
  sessionId: string
  traceId: string
  stepIndex: number
  targetTokenBudget: number
  currentTokenEstimate: number
  metadata?: Record<string, unknown>
}

interface CompressionResult {
  blocks: ContextBlock[]
  changed: boolean
  summary?: string
}

interface CompressionStrategy {
  name: string
  capabilities: Array<'trim' | 'summarize' | 'externalize' | 'restore' | 'reorder'>
  supports(ctx: CompressionContext, blocks: ContextBlock[]): boolean
  compress(ctx: CompressionContext, blocks: ContextBlock[]): Promise<CompressionResult>
}
```

### 7.3 推荐首批策略

#### `TrimStrategy`

负责：

- 裁剪超长工具结果
- 删除重复 observation
- 丢弃低优先级旧块

#### `SummarizeStrategy`

负责：

- 将较老历史压成结构化摘要块

#### `ExternalizeStrategy`

负责：

- 把大块历史写到外置存储
- 用引用块替换原块

#### `RestoreStrategy`

负责：

- 依据引用块把外置内容恢复回来

#### `ReorderStrategy`

负责：

- 调整块顺序，保证高优先级块始终靠前

---

## 8. LLM 摘要能力如何接入

### 8.1 不建议让 `ContextManager` 直接调用 provider

更合理的方式是引入：

```typescript
interface ContextSummarizer {
  summarize(args: {
    blocks: ContextBlock[]
    preserve: string[]
    maxOutputTokens: number
  }): Promise<{
    summaryBlock: ContextBlock
    sourceBlockIds: string[]
  }>
}
```

### 8.2 为什么这样更好

因为这样可以：

- 替换不同摘要模型
- 控制摘要输出格式
- 让规则压缩与 LLM 压缩并存
- 避免 `ContextManager` 和某个具体 provider 强耦合

### 8.3 摘要输出应尽量结构化

不建议直接返回一段自由文本，而建议压成结构化摘要，例如：

- 已确定目标
- 已确定约束
- 已知关键事实
- 未解决问题
- 最近关键结论

然后再由渲染层转成标准消息块。

---

## 9. 外置存储接口：`ContextArtifactStore`

### 9.1 为什么必须单独抽象

如果未来会有：

- 将大块历史写入本地
- 将超长工具输出写到文件
- 只保留引用，后续再读回来

那么这部分不能写死在压缩策略里，必须有统一存储接口。

### 9.2 推荐接口

```typescript
interface ContextArtifact {
  id: string
  type: 'raw_history' | 'tool_output' | 'summary_source'
  content: Message[]
  tokenEstimate: number
  metadata?: Record<string, unknown>
  createdAt: number
}

interface ContextArtifactStore {
  save(artifact: ContextArtifact): Promise<{ refId: string }>
  load(refId: string): Promise<ContextArtifact | null>
  delete(refId: string): Promise<void>
  search?(
    query: string,
    opts?: Record<string, unknown>
  ): Promise<Array<{ refId: string; score: number }>>
}
```

### 9.3 它支撑什么能力

- 本地文件外置
- SQLite 中间表外置
- 对象存储外置
- 后续相关性召回

---

## 10. 引用块与按需恢复

### 10.1 外置后不应该“消失”，而应变成引用块

被外置的块不应直接彻底丢掉，而应替换为一个 `external_reference` 块。

例如：

```typescript
interface ExternalReferenceMetadata {
  refId: string
  summary: string
  originalLayer: ContextLayer
}
```

这样模型仍然能看到：

- 这里曾经有一段上下文
- 这段上下文大致讲了什么
- 如果需要，可以按引用恢复

### 10.2 恢复何时发生

未来至少可能有三种恢复触发：

1. 显式恢复：用户要求查看原文
2. 相关性恢复：当前问题命中某段外置内容
3. 工具 / 计划驱动恢复：上层判断需要原文

### 10.3 推荐恢复接口

```typescript
interface RestoreQuery {
  userInput: string
  currentBlocks: ContextBlock[]
  reason: 'explicit_request' | 'relevance_hit' | 'tool_request'
}

interface ContextRestorer {
  canRestore(block: ContextBlock, query: RestoreQuery): boolean
  restore(block: ContextBlock, query: RestoreQuery): Promise<ContextBlock[]>
}
```

---

## 11. 推荐的完整工作流

未来 `ContextManager` 的完整工作流建议如下：

1. `RunEngine` 请求当前上下文快照
2. `ContextManager` 组装 `ContextBlock[]`
3. 检查预算
4. 若超预算，调用 `CompressionPolicy`
5. 根据 plan 依次执行多个 `CompressionStrategy`
6. 某些块可能被：
   - 裁剪
   - 摘要
   - 外置
   - 替换为引用
7. 再次检查预算
8. 若需要，调用 `ContextRestorer` 恢复特定引用块
9. 输出最终 `messages[]`

---

## 12. 为什么这套设计适合长期扩展

这套抽象的优点是：

- 新增一种压缩方式，不需要重写 `ContextManager`
- 新增一种外置介质，不需要改执行循环
- 新增一种恢复策略，不需要重写消息模型
- 不同业务场景可以使用不同 `CompressionPolicy`
- 系统提示词和关键约束可以在接口层硬保护

也就是说：

> 第一层不是预设某一种上下文压缩算法，而是预设一套可长期演进的压缩能力框架。

---

## 13. 首期建议落地范围

为了避免第一阶段过度设计，建议首期只落以下最小能力：

1. `ContextBlock` 模型
2. `CompressionPolicy` 抽象
3. `TrimStrategy`
4. `SummarizeStrategy` 接口（仅保留接口，不落地实现）
5. `ContextArtifactStore` 接口（仅保留接口，不落地实现）
6. `ExternalReference` 块模型（可选保留模型，不要求首期接入运行链路）

首期可以先不做：

- LLM 摘要真正执行
- 外置存储写入
- 按需恢复原文
- 自动相关性恢复
- 多种摘要模型切换
- 向量召回
- 复杂混合策略

但接口必须先留好，保证未来扩展时不需要回头重构 `ContextManager`。

---

## 14. 最终建议

建议在第一层把上下文压缩能力正式定义为：

- **分层上下文模型**
- **压缩策略管线**
- **外置存储引用**
- **按需恢复接口**

而不是：

- 一个巨大的 `compress()` 函数
- 或把所有逻辑都塞进 `ContextManager`

这会让第一层在未来面对更复杂业务时，仍然保持稳定和可扩展。

---

## 15. 可实现草案：首期最小 block 模型

首期建议先冻结如下最小 block 模型：

```typescript
type ContextLayer =
  | 'system'
  | 'pinned'
  | 'recent'
  | 'history'
  | 'memory'
  | 'tool_result'
  | 'external_reference'

type ProtectionLevel = 'immutable' | 'pinned' | 'compressible' | 'discardable'

interface ContextBlock {
  id: string
  layer: ContextLayer
  protection: ProtectionLevel
  messageCount: number
  tokenEstimate?: number
  payload: Message[]
  metadata?: Record<string, unknown>
}
```

只要这层先稳定，后面的裁剪、摘要、外置、恢复都能围绕它扩展。

## 16. 可实现草案：Phase 1 压缩管线

首期建议把压缩流程固定为下面四步：

1. `ContextBuilder` 产出按 layer 排序的 `ContextBlock[]`
2. `CompressionPolicy` 判断是否超预算
3. 若超预算，仅调用 `TrimStrategy`
4. 输出最终 `Message[]` 快照给 `ContextManager`

也就是说，Phase 1 真正落地的只有：

- block 建模
- 预算判断
- 规则裁剪

而不是完整摘要系统。

## 17. 首期只留接口、不做实现的部分

为了让第一层尽快可运行，下面这些能力建议当前只定义接口，不接入默认链路：

- `ContextSummarizer`
- `ContextArtifactStore`
- `ContextRestorer`
- `SummarizeStrategy`
- `ExternalizeStrategy`
- `RestoreStrategy`

这样既保证未来扩展不会推倒重来，也避免首期在压缩问题上过度设计。

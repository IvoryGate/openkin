# 03：面向任务系统与多 Agent 扩展的支撑设计

> 配套主文档：`00_第一层技术方案总览.md`  
> 主题：说明第一层为什么虽然只实现“单次执行原语”，但仍然能够支撑未来的任务系统、多 Agent 协作、编排、计划、投票等模式  
> 本文档角色：可实现草案中的上层依赖边界说明

---

## 1. 先说结论

第一层并不直接实现：

- 定时任务
- 多 Agent 协作
- 编排模式
- 计划模式
- 投票模式

但它必须提供这些模式共同依赖的最小运行原语。

这个原语可以概括为：

> 给定 `AgentDefinition`、`SessionRuntime` 和输入，稳定完成一次可隔离、可追踪、可扩展的 `run()`。

未来所有高级模式，本质上都是对多个 `run()` 的组织与调度。

---

## 2. 第一层到底提供了什么“原语”

第一层提供的不是业务模式，而是以下基础能力：

- 单次执行：`run()`
- 执行机制：`RunEngine`
- 执行状态：`RunState`
- 执行隔离：`Session` / `SessionRuntime`
- 执行追踪：`traceId` / `StepTrace`
- 可插拔扩展：Hook
- 外部能力调用：工具模型
- 上下文工作区：`ContextManager`
- Agent 级经验积累：`LongTermMemory`

有了这组原语，上层才能安全地组织更复杂的模式。

---

## 3. 为什么任务系统不应该下沉到第一层

任务系统的核心问题不是“怎么推理”，而是：

- 什么时候触发
- 谁来触发
- 失败如何重试
- 超时如何处理
- 状态如何持久化

这些都不是第一层关心的问题。

第一层只需要提供：

- 能执行一次 `run()`
- 能产出结果
- 能产出 trace
- 能把错误标准化

任务系统只是在第三层拿这套能力来调度。

---

## 4. 第一层如何支撑任务系统

### 4.1 任务系统真正依赖什么

任务系统最需要第一层提供三件事：

1. 明确的单次执行入口
2. 独立的执行上下文隔离
3. 可追踪的执行结果

### 4.2 一个后台任务的真实结构

后台任务不需要重写推理内核，只需要做：

1. 触发条件满足
2. 创建或选择一个 Session
3. 调用某个 Agent 的 `run()`
4. 保存结果、状态和 trace

也就是说：

```text
任务系统
  -> 准备 Session
  -> 调用 run()
  -> 记录状态
  -> 重试 / 超时 / 通知
```

### 4.3 为什么这样更稳

因为任务系统只负责编排“何时跑”，而不是篡改“怎么跑”。

这使得：

- 第一层更稳定
- 第三层任务系统更好测试
- 以后更换任务系统实现时，不会反向打破运行时内核

---

## 5. 多 Agent 协作的核心原则

### 5.1 每个 Agent 的推理现场必须隔离

多 Agent 协作时，最重要的设计原则是：

> 每个 Agent 的一次具体执行，都应运行在自己的 Session / Run 上下文里。

这意味着：

- 不共享同一个 `ContextManager`
- 不共享同一份当前 messages
- 不共享同一个 `RunState`
- 不共享同一个 run 级 trace

### 5.2 共享什么，不共享什么

#### 不共享

- 当前推理上下文
- 当前 messages
- 当前 stepIndex
- 当前工具调用链
- 当前上下文工作区

#### 可以共享或传递

- 子任务输入
- 中间结果
- 最终结果
- 投票结果
- 审核意见
- 共享执行上下文中的结构化数据

### 5.3 一句话总结

多 Agent 协作应遵循：

> 共享产物，不共享推理现场。

---

## 6. 第一层如何支撑编排模式

### 6.1 编排模式是什么

编排模式的核心不是推理，而是：

- 谁先做
- 谁后做
- 哪些任务可以并行
- 失败后怎么走

### 6.2 编排器真正做的事情

编排器应当只做这些事：

1. 生成任务图或顺序流程
2. 为每个子任务准备独立 Session
3. 调用对应 Agent 的 `run()`
4. 收集结果并决定下一个子任务

### 6.3 第一层需要提供什么支持

为了让编排模式自然成立，第一层需要保证：

- [ ] 每次 `run()` 都能独立运行
- [ ] 每个 `run()` 都有 `traceId`
- [ ] 工具系统能在每个 run 内独立工作
- [ ] `Session` 之间严格隔离
- [ ] `AgentResult` 可以被调度器直接消费

---

## 7. 第一层如何支撑计划模式

### 7.1 计划模式的本质

计划模式通常是两段式：

1. 先由 Planner 产出结构化计划
2. 再按计划驱动后续若干次执行

### 7.2 用第一层原语如何表达

它本质上只是：

- 第一次 `run()`：生成计划
- 后续多次 `run()`：执行计划中的子任务

### 7.3 第一层不需要知道“计划模式”这个业务概念

第一层只要能：

- 支持结构化输出
- 支持独立 Session
- 支持 trace

上层就可以自然实现计划模式，而无需把“Planner”“ExecutionPlan”“SubTaskDAG”这些概念压回第一层。

---

## 8. 第一层如何支撑投票模式

### 8.1 投票模式是什么

投票模式通常是：

1. 把同一个问题发给多个 Agent，或同一个 Agent 的多个独立 run
2. 收集多个候选结果
3. 再交给一个评审器或投票器做最终裁决

### 8.2 它需要第一层提供什么

- [ ] 多个 run 可以并行存在
- [ ] 每个 run 的上下文彼此独立
- [ ] 每个 run 都能得到标准化结果
- [ ] 每个 run 都可追踪

### 8.3 这说明了什么

投票模式不是新的底层运行机制，而只是上层把多个独立 `run()` fan-out 之后再聚合。

---

## 9. 为什么未来高级模式都不应该共享同一个 ContextManager

如果多个 Agent 共用同一个 `ContextManager`，会立刻出现这些问题：

- 谁写入了什么难以追踪
- 噪音上下文互相污染
- 审计和回放困难
- 权限边界模糊
- 并行执行时状态容易踩踏

因此应坚持：

- 每个 Agent 的每次执行都有自己的上下文工作区
- 每次执行都拥有自己的 `RunState`
- 协作层只在它们之间搬运结构化结果

这比“大家共用一个大上下文池”更可控，也更适合长期演进。

---

## 10. 第一层应预留哪些扩展点

为了确保未来不返工，第一层至少要预留以下能力：

### 10.1 `sessionType` 可扩展

未来可能会出现：

- `user_conversation`
- `agent_task`
- `scheduled_task`
- `review_task`
- `vote_round`

因此 Session 类型不能只写死当前几种。

但为了避免上层随意注入未经约束的新类型，扩展方式应采用：

- 注册式扩展

而不是：

- 任意开放字符串直接进入运行时

### 10.2 `RunContext.metadata`

上层调度器需要透传额外信息，例如：

- `parentTaskId`
- `executionId`
- `plannerTraceId`
- `voteRoundId`
- `schedulerHints`

第一层不解释这些字段，但必须允许携带。

### 10.3 `AgentResult` 必须标准化

上层模式都要依赖统一结果结构：

- `traceId`
- `finalMessage`
- `finishReason`
- `steps`

### 10.4 Hook 必须稳定

任务系统、多 Agent 协作、审计、安全都可能需要在不同阶段挂能力，因此 Hook contract 必须稳定。

### 10.5 工具模型必须稳定

未来第二层和第三层提供的工具来源会越来越多，但第一层的工具调用模型必须尽量不变。

### 10.6 上下文压缩与外置策略必须可插拔

未来高级模式可能需要：

- 长链路历史外置存储
- 按投票回合恢复原始片段
- 按计划节点召回某段旧上下文

因此上下文压缩与恢复能力必须在第一层就被设计为可插拔体系。

---

## 11. 是否需要把 `RunEngine` 抽象成接口

我的建议是：**是，尽早抽象。**

### 原因

虽然第一阶段只有一个默认执行引擎，但未来很可能出现：

- `ReActRunEngine`
- `SafeModeRunEngine`
- `PlanExecuteRunEngine`
- `ReviewRunEngine`

如果现在把默认执行逻辑写死成唯一真理，未来就会变成在同一个类里不断堆分支。

### 建议

第一层至少预留：

```typescript
interface RunEngine {
  run(args: RunArgs): Promise<AgentResult>
}
```

第一阶段只实现：

- `ReActRunEngine`

这样未来扩展策略时，底层就不需要硬拆。

---

## 12. 与未来层级的接口关系

### 12.1 第一层与第三层

第三层负责：

- 定时任务
- 服务接口
- 持久化
- 监控

第一层负责：

- 单次运行原语

两者关系是：

> 第三层组织、持久化和服务化第一层的运行原语。

### 12.2 第一层与第四层

第四层负责：

- 子任务拆解
- DAG 调度
- Supervisor / Worker 协作
- 计划、投票、编排等策略

第一层负责：

- 提供每个 Agent 的独立执行能力

两者关系是：

> 第四层调度多个第一层 `run()`，而不是重写第一层执行机制。

---

## 13. 设计原则总结

为了让第一层真正支撑未来扩展，本节总结必须坚持的原则：

1. 第一层只提供单次执行原语，不直接实现高级协作模式。
2. 多 Agent 协作时，每个 Agent 的执行现场必须隔离。
3. 协作层共享的是任务、结果和结构化执行信息，而不是同一个上下文工作区。
4. 任务系统解决“何时跑”，第一层解决“如何稳定跑完一次”。
5. 未来高级模式应建立在多个 `run()` 之上，而不是侵入单次执行内核。
6. `RunEngine` 应尽早抽象为接口，以便未来容纳不同执行策略。
7. `RunState` 只代表一次执行的状态，不应和 Session 级资源混用。

---

## 14. 可实现草案：上层真正依赖什么

为了让第三层和第四层能直接开工，第一层至少应稳定输出这些上层可依赖原语：

```typescript
interface AgentFacade {
  run(input: AgentRunInput, options?: RunOptions): Promise<AgentResult>
}

interface AgentResult {
  traceId: string
  sessionId: string
  status: RunFinalStatus
  output?: Message
  steps: StepTrace[]
  error?: RunError
}
```

上层真正依赖的是：

- 可重复调用的 `run()`
- 可透传的 `RunOptions`
- 可直接消费的 `AgentResult`
- 标准化的终态与错误语义

而不是第一层内部的 `ContextManager`、`ToolProvider`、`HookRunner` 细节。

## 15. 可实现草案：对任务系统和多 Agent 的最低支撑

只要第一层具备下面四点，上层就已经能构建任务系统和多 Agent 编排：

1. 同一个 `AgentDefinition` 能在不同 `Session` 中独立运行
2. 一次 `run()` 有唯一 `traceId`
3. `AgentResult.status` 能区分完成、失败、取消、预算耗尽
4. `RunContext.metadata` 或等价字段可以透传调度信息

这意味着首期不需要实现任何高级协作模式，就能先把“上层能不能接”这个问题解决掉。

## 16. 首期不应下沉到第一层的能力

以下能力即使未来一定会做，也不建议在第一层首期落地：

- DAG 调度器
- Planner / Supervisor 角色编排
- 投票聚合器
- 重试状态机
- 分布式任务恢复

第一层只需要保证：

> 任何上层系统都能把自己的调度决策收敛成一次次稳定的 `run()` 调用。

## 4. 核心对象关系图

这张图回答的是：**几个最容易混淆的对象，到底谁是静态定义、谁是运行态、谁是隔离容器。**

```mermaid
classDiagram
    class AgentDefinition {
      +id
      +name
      +systemPrompt
      +llmProfile
      +maxSteps
      +pluginConfigs
    }

    class Session {
      +id
      +agentId
      +sessionType
      +initiator
      +createdAt
      +updatedAt
    }

    class SessionRuntime {
      +session
      +contextManager
      +runtimeCache
    }

    class AgentRun {
      +traceId
      +runInput
      +runContext
      +stepIndex
      +stepTraces
      +result
    }

    class ContextManager {
      +loadBaseMessages()
      +append()
      +snapshot()
      +ensureWithinBudget()
    }

    AgentDefinition --> AgentRun : used by
    Session --> SessionRuntime : materializes to
    SessionRuntime --> ContextManager : owns
    SessionRuntime --> AgentRun : participates in
    AgentDefinition --> Session : many sessions can target same agent
```

一句话概括这张图：

- `AgentDefinition` 是“谁”
- `Session` 是“属于谁”
- `SessionRuntime` 是“这次会话在内存里有什么”
- `AgentRun` 是“这次执行正在发生什么”

## 6. 一次 `run()` 的核心时序图

这张图回答的是：**从用户输入到拿到最终结果，中间的核心时序是什么。**

```mermaid
sequenceDiagram
    autonumber
    participant Caller as 调用方
    participant Registry as SessionRegistry
    participant Runtime as SessionRuntime
    participant Runner as AgentRunner
    participant Hooks as HookRunner
    participant Context as ContextManager
    participant LLM as LLMProvider
    participant Tool as ToolExecutor

    Caller->>Registry: 获取 SessionRuntime(sessionId)
    Registry-->>Caller: SessionRuntime
    Caller->>Runner: run(agentDefinition, sessionRuntime, input)
    Runner->>Hooks: onRunStart(ctx)
    Hooks-->>Runner: done

    loop step < maxSteps
        Runner->>Hooks: onContextBuild(stepCtx)
        Hooks-->>Runner: ContextAddition[]
        Runner->>Context: build messages
        Runner->>Hooks: onBeforeLLMCall(stepCtx, messages)
        Hooks-->>Runner: final messages
        Runner->>LLM: invoke(messages, tools)
        LLM-->>Runner: LLMResponse
        Runner->>Hooks: onAfterLLMCall(stepCtx, response)
        Hooks-->>Runner: final response

        alt 有工具调用
            loop each tool call
                Runner->>Hooks: onBeforeToolCall(stepCtx, toolCall)
                Hooks-->>Runner: continue / abort / replace
                Runner->>Tool: execute(toolCall)
                Tool-->>Runner: ToolResult
                Runner->>Hooks: onAfterToolCall(stepCtx, result)
                Hooks-->>Runner: final ToolResult
                Runner->>Context: append tool result
            end
            Runner->>Hooks: onStepEnd(stepCtx, summary)
        else 无工具调用，得到最终答案
            Runner->>Hooks: onStepEnd(stepCtx, summary)
            Runner->>Hooks: onRunEnd(ctx, result)
            Runner-->>Caller: AgentResult
        end
    end

    Runner->>Hooks: onRunError(ctx, error)
    Runner-->>Caller: throw error
```

阅读重点：

- Hook 不只是前后包一层，而是插在多个关键点
- 每次 LLM 调用前都要重新构建 context
- 工具结果不是直接结束，而是会被回注到 context 中继续推理

## 7. Hook 插入点总览图

这张图回答的是：**第一层为什么一定要内建 Hook，以及每个 Hook 的位置在哪里。**

```mermaid
flowchart TB
    A["run() start"]
    B["onRunStart"]
    C["build context"]
    D["onContextBuild"]
    E["onBeforeLLMCall"]
    F["LLM invoke / stream"]
    G["onAfterLLMCall"]
    H{"has tool calls?"}
    I["onBeforeToolCall"]
    J["execute tool"]
    K["onAfterToolCall"]
    L["append tool result to context"]
    M["onStepEnd"]
    N{"final answer?"}
    O["onRunEnd"]
    P["onRunError"]

    A --> B --> C --> D --> E --> F --> G --> H
    H -- Yes --> I --> J --> K --> L --> M --> N
    H -- No --> M --> N
    N -- Yes --> O
    N -- No --> C

    B -.异常不应中断主流程（观察型）.-> P
    E -.变换失败.-> P
    G -.变换失败.-> P
    I -.abort / replace.-> P
    J -.工具异常.-> P
```

阅读重点：

- `onContextBuild` 和 `onBeforeLLMCall` 不是一回事
- `onAfterLLMCall` 是工具执行前最后一个“全局拦截点”
- `onBeforeToolCall` 是每个工具调用级别的最后一道门

## 8. Context、History、LongTermMemory 的数据流向图

这张图回答的是：**三个最容易搅在一起的模块，分别在什么时候参与，数据怎么流。**

```mermaid
flowchart TB
    subgraph SessionScope["Session 级数据"]
        HIST["ConversationHistory"]
        CM["ContextManager"]
    end

    subgraph AgentScope["Agent 级数据"]
        LTM["LongTermMemory"]
    end

    subgraph RunLoop["单次运行"]
        INPUT["本次输入"]

        LLM["LLM Call"]
        RESULT["最终结果"]
    end

    HIST -->|onRunStart: 加载历史| CM

    CM --> BUILD
```

阅读重点：

- `ConversationHistory` 是按 `sessionId` 的“档案室”
- `LongTermMemory` 是按 `agentId` 的“经验库”
- `ContextManager` 是本次运行给 LLM 的“工作台”

## 12. 并发安全图

这张图回答的是：**哪些对象如果放错位置，就会产生串台和并发污染。**

```mermaid
flowchart TB
    subgraph Safe["正确放置"]
        S1["AgentDefinition: 静态配置"]
        S2["SessionRuntime: ContextManager"]
        S3["AgentRun: stepIndex / stepTraces / usage"]
        S4["Plugin State: Map<traceId, state>"]
    end

    subgraph Unsafe["错误放置"]
        U1["Agent 单例上挂当前 messages"]
        U2["Agent 单例上挂当前 stepIndex"]
        U3["全局变量存当前 trace"]
        U4["插件实例只存 lastRunState"]
    end

    Safe --> OK["可并发运行多个 Session"]
    Unsafe --> BAD["高风险：上下文串台 / trace 污染 / 状态泄漏"]
```

阅读重点：

- 第一层并发问题本质上是“运行态对象放错层”
- 解决办法不是引入复杂锁，而是把生命周期和所有权设计对

## 15. 总结

这份图册的核心目的不是把设计“画得更漂亮”，而是把几个最容易口头说混的点变成可视化结构：

- 第一层是运行时内核，不是服务层
- `AgentDefinition`、`SessionRuntime`、`AgentRun` 必须分层
- `ContextManager` 必须属于 Session
- Hook 是上层能力接入第一层的标准接口
- 记忆和持久化必须通过 port 隔离
- 并发安全的关键不在锁，而在所有权与生命周期

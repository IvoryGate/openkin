# 06：错误与取消语义模型

> 配套主文档：`00_第一层技术方案总览.md`  
> 主题：为第一层定义稳定的失败语义、取消语义、超时语义和异常边界，避免上层系统各自发明一套状态机  
> 本文档角色：可实现草案中的错误 contract、状态流转与默认处理规则

---

## 1. 为什么必须单独建模

如果第一层没有统一错误模型，上层最终会遇到这些问题：

- 任务系统无法判断是否该重试
- 编排系统无法判断某个 Agent 是失败、取消还是主动终止
- 监控系统无法统一统计失败类型
- 多 Agent 协作无法稳定传播失败原因

所以第一层不能只“抛异常”，必须定义：

- 标准错误码
- 标准终止原因
- 标准取消信号
- 标准超时语义

---

## 2. 需要区分的几类终止结果

一次 `run()` 结束时，至少要能区分下面几类结果：

### 2.1 正常完成

表示：

- 得到了最终回复
- 或完成了预期工具链路

### 2.2 业务性中止

表示：

- 命中了安全规则
- 权限校验拒绝
- 人工审核不通过
- 上层策略要求提前停止

这类终止不一定是“系统错误”。

### 2.3 主动取消

表示：

- 用户点击取消
- 上层任务系统取消
- 父任务取消导致子运行停止

### 2.4 预算耗尽

表示：

- 达到最大 step 数
- 达到 token 预算
- 达到工具调用次数预算

### 2.5 运行失败

表示：

- LLM 调用失败
- 工具执行失败
- 上下文构建失败
- 未知运行时异常

---

## 3. 推荐的顶层结果模型

```typescript
type RunFinalStatus =
  | 'completed'
  | 'aborted'
  | 'cancelled'
  | 'budget_exhausted'
  | 'failed'
```

建议 `AgentResult` 至少包含：

```typescript
interface AgentResult {
  traceId: string
  sessionId: string
  status: RunFinalStatus
  output?: Message
  steps: StepResult[]
  finishReason?: string
  error?: RunError
  usage?: UsageStats
}
```

说明：

- `status` 描述最终状态
- `finishReason` 描述更细粒度原因
- `error` 只在需要表达失败细节时出现

---

## 4. 错误码应如何分层

推荐分三层错误码：

### 4.1 `LLMErrorCode`

用于模型层错误。

例如：

- `LLM_RATE_LIMIT`
- `LLM_TIMEOUT`
- `LLM_UNAVAILABLE`
- `LLM_INVALID_RESPONSE`
- `LLM_CONTEXT_OVERFLOW`

### 4.2 `ToolErrorCode`

用于工具层错误。

例如：

- `TOOL_NOT_FOUND`
- `TOOL_INVALID_INPUT`
- `TOOL_EXECUTION_FAILED`
- `TOOL_TIMEOUT`
- `TOOL_PERMISSION_DENIED`

### 4.3 `RunErrorCode`

用于运行编排层错误。

例如：

- `RUN_CANCELLED`
- `RUN_ABORTED`
- `RUN_MAX_STEPS_EXCEEDED`
- `RUN_TOKEN_BUDGET_EXCEEDED`
- `RUN_CONTEXT_BUILD_FAILED`
- `RUN_HOOK_FAILED`
- `RUN_INTERNAL_ERROR`

---

## 5. 推荐错误对象

```typescript
interface RunError {
  code: string
  message: string
  retryable?: boolean
  source: 'llm' | 'tool' | 'runtime' | 'hook' | 'context'
  details?: Record<string, unknown>
  cause?: unknown
}
```

说明：

- `code` 负责机器判断
- `message` 负责人类可读
- `retryable` 负责上层调度判断
- `source` 负责观测和归因

---

## 6. 取消语义应该怎么接入

推荐第一层统一对接标准取消信号：

```typescript
interface RunOptions {
  abortSignal?: AbortSignal
  timeoutMs?: number
  maxSteps?: number
  maxToolCalls?: number
  maxPromptTokens?: number
}
```

### 取消的传播路径

1. 上层传入 `abortSignal`
2. `RunEngine` 在 step 边界检查取消状态
3. 正在执行的 LLM / Tool 调用尝试透传取消信号
4. `RunState` 进入 `cancelled` 终态

### 为什么不建议自己定义另一套取消 token

因为：

- Node.js 生态已有 `AbortSignal`
- fetch、部分 SDK、未来更多底层库都原生支持
- 第三层任务系统更容易统一接入

因此推荐：

> 第一层对外使用 `AbortSignal`，内部不要再发明一套并行取消机制。

---

## 7. 超时和取消不是一回事

必须显式区分：

### 7.1 取消

来自外部意图：

- 用户取消
- 父任务取消
- 调度器取消

### 7.2 超时

来自预算策略：

- LLM 调用超时
- Tool 调用超时
- 整体 run 超时

因此超时更适合作为：

- `failed`
- 或 `budget_exhausted`

而不应默认归类成 `cancelled`。

---

## 8. 哪些错误应该抛异常，哪些应该进入 `AgentResult`

推荐边界如下：

### 8.1 预期内终止

例如：

- 用户取消
- 预算耗尽
- 业务规则拒绝

建议：

- 不抛出未分类异常
- 直接收敛到 `AgentResult.status`

### 8.2 非预期失败

例如：

- 模型返回畸形结构
- 工具执行器崩溃
- Hook 内部未处理异常

建议：

- 由内核统一捕获
- 标准化成 `RunError`
- 最终返回带 `failed` 状态的 `AgentResult`

### 8.3 只有框架边界错误才直接抛出

例如：

- 调用参数完全非法
- AgentDefinition 缺失必填字段
- 编程错误导致无法启动运行

这类属于“调用方式错误”，可以直接抛出。

---

## 9. Hook、Tool、LLM 的失败边界

### 9.1 Hook

Hook 应分类型处理：

- 观察型 Hook 失败：默认记录并忽略
- 变换型 Hook 失败：默认视为运行失败
- 拦截型 Hook 返回拒绝：属于业务性中止，不一定是系统错误

### 9.2 Tool

工具失败要区分：

- 不存在
- 输入不合法
- 权限被拒绝
- 执行异常
- 超时

不要全部折叠成 `TOOL_EXECUTION_FAILED`。

### 9.3 LLM

模型失败至少区分：

- 供应商错误
- 超时
- 上下文溢出
- 响应结构非法

---

## 10. 与任务系统和多 Agent 的关系

上层任务系统需要利用第一层的统一失败语义做这些事：

1. 判断是否重试
2. 判断是否切换模型
3. 判断是否降级到人工处理
4. 判断是否传播取消给下游 Agent

多 Agent 协作也要依赖这一点：

- Planner 失败，不等于 Worker 失败
- 某个 Worker 被取消，不等于整轮协作失败
- Vote 阶段失败，可能只需重跑一部分 Agent

如果第一层没有统一语义，上层编排会非常脆弱。

---

## 11. 推荐的首期落地范围

首期建议必须落地：

1. `RunFinalStatus`
2. `RunError`
3. `LLMErrorCode`
4. `ToolErrorCode`
5. `RunErrorCode`
6. `RunOptions.abortSignal`
7. `timeoutMs / maxSteps / maxToolCalls`

首期可以暂不细化：

- 复杂重试策略
- 分布式取消传播
- 错误码国际化
- 细粒度 SLA 分类

---

## 12. 最终建议

建议把第一层错误模型冻结为三个原则：

1. **终态统一**：所有 `run()` 都收敛到有限状态集合
2. **错误分层**：LLM、Tool、Runtime 错误码分层定义
3. **取消标准化**：统一接 `AbortSignal`，不要另造协议

这样第三层任务系统、第四层协作系统、第五层业务系统才能稳定依赖第一层的失败语义，而不是在上层各自补洞。

---

## 13. 可实现草案：推荐的错误对象与基类

首期实现时建议不要让系统里到处散落原生 `Error`，而是统一收敛到一层基础错误：

```typescript
class AgentRuntimeError extends Error {
  constructor(
    public readonly code: string,
    message: string,
    public readonly source: 'llm' | 'tool' | 'runtime' | 'hook' | 'context',
    public readonly retryable: boolean = false,
    public readonly details?: Record<string, unknown>,
  ) {
    super(message)
  }
}
```

然后由：

- LLM 适配层产出 `LLM_*`
- 工具层产出 `TOOL_*`
- 执行引擎层产出 `RUN_*`

最终统一映射成 `RunError` 挂到 `AgentResult.error`。

## 14. 可实现草案：状态流转约束

首期建议把 `RunState.status` 的状态流转限制为：

```text
running
  ├─ completed
  ├─ aborted
  ├─ cancelled
  ├─ budget_exhausted
  └─ failed
```

约束要点：

- 一次 `run()` 只能进入一个终态
- 进入终态后不得再继续 step 循环
- `cancelled` 只用于外部取消信号
- `budget_exhausted` 只用于预算类终止
- `failed` 只用于非预期失败

## 15. 首期默认处理规则

为了避免实现时每个人各写一套规则，建议首期默认约定为：

1. 观察型 Hook 报错：记录并忽略
2. 变换型 Hook 报错：转为 `RUN_HOOK_FAILED`
3. Tool 超时：转为 `TOOL_TIMEOUT`
4. LLM 上下文溢出：转为 `LLM_CONTEXT_OVERFLOW`
5. 外部 `AbortSignal` 触发：转为 `cancelled`
6. `maxSteps` / `maxToolCalls` / `maxPromptTokens` 命中：转为 `budget_exhausted`

这些规则先稳定下来，第三层任务系统和观测系统才有统一行为可依赖。

## 1. 文档目标

本方案只聚焦一件事：**把“第一层：通用 Agent 基础框架”定义成一套能够长期承接上层业务与未来扩展的稳定底座。**

它要解决的不是“如何快速跑通一个 Demo”，而是以下更基础的问题：

- 如何定义一套不会被第二到第五层反复打破的第一层 contract
- 如何把 `Session`、`run()`、`step`、`RunState` 这些概念彻底拆开
- 如何让执行引擎、上下文管理、工具调用、记忆、Hook、执行轨迹在职责上闭环
- 如何为未来的任务系统、多 Agent 协作、计划模式、投票模式、编排模式预留稳定接口
- 如何让上下文压缩、外置存储、按需恢复等高级策略在第一层自然扩展，而不需要回头重构

### 1.1 本轮“可实现草案”的含义

本轮文案不再只回答“设计应该是什么”，而要进一步回答“工程上应该怎样拆、首期应该先落什么、哪些接口要先冻结”。

因此本版文档至少要做到：

- 给出可直接建文件的模块边界
- 给出首期必须冻结的类型 contract
- 给出默认实现的最小执行闭环
- 给出 Phase 1 与后续扩展的清晰分界
- 让第二层到第五层可以直接依赖第一层类型，而不是继续猜测语义

## 2. 第一层的职责与边界

### 2.1 职责

第一层是**Agent 运行时内核**，负责定义单个 Agent 一次执行所需的全部基础原语。

它负责：

- 定义统一的 LLM 访问抽象
- 定义消息模型、工具调用模型、流式响应模型
- 定义 `Session`、`SessionRuntime`、`run()`、`step`、`RunState` 的运行时关系
- 驱动单 Agent 的默认执行引擎
- 提供生命周期 Hook 扩展体系
- 提供上下文构建、预算控制、压缩策略编排、工具执行、记忆接入、执行轨迹记录的标准边界

### 2.2 不在职责内

第一层不负责：

- HTTP / WebSocket 服务
- SQLite / Drizzle / 迁移脚本
- MCP 协议接入
- Skill 动态加载
- 定时任务、Cron、重试状态机
- 多 Agent 调度与协作策略
- 监控平台、日志平台、告警平台的具体实现

### 2.3 完成标志

第一层完成后，系统应当能够在**无 HTTP 服务、无具体 DB 实现、无上层调度器**的前提下，通过代码直接完成：

1. 创建一个 Agent 定义；
2. 在指定 Session 中发起一次 `run()`；
3. 通过 LLM 与工具完成若干步推理；
4. 输出最终结果、轨迹与错误信息；
5. 允许通过 Hook 接入历史、长期记忆、日志、安全、压缩策略等能力。

## 3. 核心设计结论

### 3.1 五个必须分开的概念

第一层必须明确区分以下五类对象：

| 对象 | 作用 | 生命周期 |
| --- | --- | --- |
| `AgentDefinition` | Agent 的静态身份与默认配置 | 长生命周期 |
| `Session` | 上下文归属与隔离单位 | Session 级 |
| `SessionRuntime` | Session 在内存中的运行时资源 | Session 级 |
| `RunEngine` | 一次执行所采用的运行机制 | 长生命周期 / 可复用 |
| `RunState` | 一次具体执行的运行态状态对象 | Run 级 |

### 3.2 `run()` 不等于 `Session`

一个 `Session` 可以包含多次 `run()`。

例如用户在同一个对话窗口连续发送三条消息：

- 第一次发消息 -> 第一次 `run()`
- 第二次发消息 -> 第二次 `run()`
- 第三次发消息 -> 第三次 `run()`

因此关系是：

```text
Session
  ├─ Run #1
  ├─ Run #2
  └─ Run #3
```

### 3.3 `RunState` 不等于所有运行时对象

`RunState` 是一次执行的状态视图或状态容器，它会引用本次执行所依赖的 `SessionRuntime`、`ContextManager` 等资源，但不等于这些资源本身。

换句话说：

- `SessionRuntime` 是执行环境
- `RunState` 是本次执行在这个环境中的状态

### 3.4 `ContextManager` 属于 Session，而不是 Agent

`ContextManager` 是当前 Session 的上下文工作区，而不是 `AgentDefinition` 的成员。

原因：

- 同一个 Agent 可能同时服务多个 Session
- 如果 `ContextManager` 挂在 Agent 上，会天然制造串台风险
- Session 关闭或失活时，对应运行时资源应被回收

### 3.5 第一层真正提供的是“单次执行原语”

第一层对上层提供的核心能力，不是某种具体协作模式，而是：

> 给定 `AgentDefinition`、`SessionRuntime` 和输入，稳定完成一次可隔离、可追踪、可扩展的执行。

未来的任务系统、多 Agent 协作、计划模式、投票模式、编排模式，本质上都是对多个 `run()` 的组织，而不是对第一层执行内核的重写。

### 3.6 可实现冻结范围

当前版本建议先冻结以下 contract，再进入编码：

1. 运行时基础类型：`AgentRunInput`、`RunOptions`、`RunState`、`AgentResult`
2. LLM 抽象：`LLMProvider`、`StreamChunk`、`ToolCall`
3. Session 体系：`Session`、`SessionRuntime`、`SessionRegistry`
4. 上下文体系：`ContextManager`、`ContextBuilder`、`ContextBlock`
5. 工具体系：`ToolDefinition`、`ToolProvider`、`ToolRuntimeView`、`ToolExecutor`
6. Hook 体系：`AgentLifecycleHook`、`HookRunner`
7. 错误体系：`RunError`、`LLMErrorCode`、`ToolErrorCode`、`RunErrorCode`

这些类型一旦冻结，首期就可以直接开始落最小执行闭环。

## 4. 第一层总体结构

```text
第一层
├─ 运行时统一契约
├─ LLM 抽象层
├─ Session 与 SessionRuntime
├─ RunEngine 与 RunState
├─ ContextManager 与 ContextBuilder
├─ 工具模型与工具运行时
├─ 生命周期 Hook 系统
├─ 记忆系统与存储端口
├─ 执行轨迹与可观测边界
└─ 上下文压缩 / 外置 / 恢复扩展接口
```

这几部分之间的关系是：

- `AgentDefinition` 描述“谁在执行”
- `SessionRuntime` 提供“在哪个隔离上下文里执行”
- `RunEngine` 决定“怎么执行”
- `RunState` 表示“这次执行当前的状态”
- `ContextManager` 负责“本次发送给 LLM 的上下文工作区”
- 工具运行时负责“哪些工具可见、如何执行”
- Hook 系统负责“上层如何接入而不侵入核心”

## 5. 分模块技术方案

### 5.1 运行时统一契约

第一层必须先冻结统一 contract，后续所有文档和上层模块只能围绕这组 contract 扩展，不能绕开它另起一套字段。

#### 核心类型

- [ ] `AgentRunInput`
- [ ] `RunContext`
- [ ] `StepContext`
- [ ] `RunState`
- [ ] `AgentResult`
- [ ] `StepTrace`
- [ ] `Message`
- [ ] `ToolCall`
- [ ] `ToolResult`
- [ ] `StreamChunk`

#### 关键要求

- [ ] `run()` 必须显式接收 `sessionId`
- [ ] `RunContext` 必须显式携带 `traceId`
- [ ] `RunState` 必须显式表达本次执行的步骤、状态、结果和错误
- [ ] `AgentResult` 必须携带 `traceId` 和 `steps`
- [ ] `metadata` 允许上层透传扩展信息，但第一层不解释业务语义

> 详细说明见：[`01_运行时对象与执行引擎.md`](./01_运行时对象与执行引擎.md)

### 5.2 LLM 抽象层

第一层的 LLM 抽象层只向上暴露稳定语义，不向上泄露 provider 差异。

#### 能力目标

- [ ] 统一同步调用：`invoke()`
- [ ] 统一流式调用：`stream()`
- [ ] 统一 token 统计：`countTokens()`
- [ ] 统一模型能力描述：`getModelInfo()`

#### 设计要求

- [ ] `Message` 模型从第一天起支持文本、图片、工具结果、结构化内容等多种内容分片
- [x] `Message.content` 预留 `file` / `audio` 类型内容分片，首期 contract 即保留
- [ ] provider adapter 负责吸收 OpenAI / Anthropic / Gemini 等差异
- [ ] 第一层不直接暴露 provider 私有字段给上层
- [ ] LLM 响应中统一抽象 `toolCalls`、`usage`、`finishReason`

### 5.3 Session 与 SessionRuntime

#### Session 的定位

`Session` 是**上下文归属单位**，不是一次执行本身。

它回答的是：

- 这段历史属于谁
- 这次交互属于哪个对话窗口 / 子任务 / 调度任务
- 当前上下文应与谁隔离

#### SessionRuntime 的定位

`SessionRuntime` 是 Session 在内存中的资源容器，负责持有：

- [ ] `ContextManager`
- [ ] Session 级缓存
- [ ] 与当前 Session 绑定的辅助资源

#### 设计要求

- [ ] `Session` 和 `run()` 必须严格分离
- [ ] `SessionRuntime` 与 `Session` 一一对应
- [ ] `ContextManager` 必须属于 `SessionRuntime`
- [ ] 同一个 Agent 的多个 Session 必须互相隔离
- [x] `sessionType` 仅允许注册式扩展，不允许任意开放字符串

### 5.4 RunEngine、RunState 与执行循环

#### RunEngine 的定位

`RunEngine` 是执行引擎，是“怎么跑一次执行”的机制抽象。

第一阶段的明确决策是：

- [x] `RunEngine` 采用接口化设计
- [x] 默认实现为 `ReActRunEngine`

未来可扩展为：

- `PlanExecuteRunEngine`
- `SafeModeRunEngine`
- `ReviewRunEngine`

#### RunState 的定位

`RunState` 是一次 `run()` 的运行态状态对象，负责承载：

- [ ] `traceId`
- [ ] `sessionId`
- [ ] 当前 `stepIndex`
- [ ] 本次输入
- [ ] 已记录 `StepTrace[]`
- [ ] 当前状态与错误
- [ ] 最终结果

#### 设计要求

- [ ] `RunEngine` 负责流程控制，不负责会话归属
- [ ] `RunState` 负责本次执行状态，不应膨胀为“所有运行时对象的大集合”
- [ ] `RunEngine` 应使用 `SessionRuntime`，而不是把 Session 级资源复制到 `RunState`
- [ ] 所有 run 级状态必须隔离，不能挂在 `AgentDefinition` 上

> 详细说明见：[`01_运行时对象与执行引擎.md`](./01_运行时对象与执行引擎.md)

### 5.5 ContextManager、ContextBuilder 与预算控制

#### ContextManager 的职责

`ContextManager` 不是数据库，不是调度器，也不是摘要算法集合；它是当前 Session 的上下文工作区与预算控制器。

它负责：

- [ ] 维护当前候选上下文块
- [ ] 生成当前要发送给 LLM 的消息快照
- [ ] 检查 token 预算
- [ ] 调用压缩 / 外置 / 恢复策略管线
- [ ] 保证不可压缩信息始终被保留

#### 首次调用前的消息装配

首次 `run()` 的第一个 step 中，执行引擎应完成以下动作：

1. 从 `AgentDefinition` 读取 `systemPrompt`
2. 从会话历史恢复该 `sessionId` 对应消息
3. 追加本次用户输入
4. 通过 `onContextBuild` 收集长期记忆 / RAG / 其他上下文贡献
5. 交给 `ContextManager` 生成候选上下文
6. 应用预算与压缩策略
7. 将最终 `messages` 交给 LLM

#### 后续步骤的上下文重建

如果本 step 触发了工具调用，工具结果会回注到 `ContextManager`，下一次 step 再重新生成新的 `messages`。

因此第一层不是“首次装配一次上下文后一直复用”，而是：

- 首步前完成首次装配
- 每一步前重新生成上下文快照

#### 上下文压缩的职责边界

`ContextManager` 应负责压缩策略的**编排与预算控制**，但不应独自实现所有压缩算法。

建议拆成：

- `CompressionPolicy`
- `CompressionStrategy`
- `ContextSummarizer`
- `ContextArtifactStore`
- `ContextRestorer`

#### 系统提示词保护

第一层必须保证：

- [ ] `systemPrompt` 位于不可压缩层
- [ ] 关键约束与安全规则具有固定保留预算
- [ ] 历史摘要不得替代系统层语义

> 详细说明见：[`02_上下文_会话_记忆_工具.md`](./02_上下文_会话_记忆_工具.md)  
> 压缩与恢复扩展见：[`04_上下文压缩_外置存储与恢复策略.md`](./04_上下文压缩_外置存储与恢复策略.md)

#### 首期压缩策略决策

- [x] 首期仅实现规则剪裁（`TrimStrategy`）
- [x] LLM 摘要、外置存储、按需恢复仅保留接口，不在首期实现

### 5.6 工具模型与工具运行时

工具问题必须拆成三个阶段：

#### A. 工具定义

- [ ] 统一 `ToolSchema`
- [ ] 统一 `ToolCall`
- [ ] 统一 `ToolResult`

#### B. 工具注册

这是**装配期行为**，而不是执行循环里的行为。

在 Agent 初始化或能力装配阶段，应先确定：

- [ ] 该 Agent 有哪些可用工具
- [ ] 这些工具来自哪里（内置 / Skill / MCP / 上层注入）
- [ ] 对外暴露哪些 schema

#### C. 工具使用

这是**运行期行为**。

在每次调用 LLM 之前：

- [ ] `RunEngine` 从工具运行时获取当前可用工具 schema
- [ ] 将 schema 作为 `tools` 注入给 LLM

在 LLM 返回 `toolCalls` 后：

- [ ] `RunEngine` 按名字路由到对应工具
- [ ] 执行工具
- [ ] 将工具结果回注到上下文

#### 设计要求

- [ ] 工具注册和工具执行必须明确分离
- [ ] 工具运行时必须支持按 Agent 能力筛选工具
- [ ] 第二层的 MCP / Skill 只是在第一层工具模型之上扩展来源

> 正式接口组见：[`05_工具运行时接口组.md`](./05_工具运行时接口组.md)

### 5.7 生命周期 Hook 系统

#### 为什么必须有 Hook

因为第一层不是把所有能力都做进核心，而是要让：

- 历史恢复
- 记忆注入
- 安全拦截
- 审计日志
- 轨迹记录
- 任务系统联动

都能通过标准扩展点接入。

#### 谁触发，谁执行

生命周期 Hook 的“触发者”是 `RunEngine`，但“做事情的人”是插件。

例如：

- `onRunStart` 由执行引擎触发
- `ConversationHistoryPlugin` 在 `onRunStart` 中加载历史
- `TraceLoggerPlugin` 在 `onAfterLLMCall` 中记录轨迹
- `ToolPermissionGuard` 在 `onBeforeToolCall` 中做权限检查

#### Hook 分类

- [ ] 观察型：日志、审计、监控、轨迹
- [ ] 变换型：改写消息、改写 LLM 响应、改写工具结果
- [ ] 拦截型：权限、熔断、替换输入、中止执行

#### 设计要求

- [ ] Hook 时机必须稳定
- [ ] 观察型异常不影响主流程
- [ ] 变换型按注册顺序串行处理
- [ ] 拦截型允许 `continue / abort / replace`
- [ ] 核心层只认识 Hook 接口，不认识任何具体插件实现

### 5.8 记忆系统与存储端口

第一层需要记忆能力，但不应直接绑定 SQLite。

#### 两类记忆

##### ① ConversationHistory

- [ ] 按 `sessionId` 读写会话消息
- [ ] 在 `onRunStart` 恢复历史
- [ ] 在 `onRunEnd` 持久化新增消息

##### ② LongTermMemory

- [ ] 按 `agentId` 保留跨 Session 经验
- [ ] 在首步 `onContextBuild` 注入相关记忆
- [ ] 在 `onRunEnd` 提炼本轮经验

#### 为什么第一层只定义 port

如果第一层直接写死 SQLite，未来：

- 第三层数据库实现会侵入核心
- 记忆能力无法替换
- 测试时难以 mock

因此第一层只应定义：

- [ ] `ConversationStore`
- [ ] `MemoryStore`
- [ ] `ContextArtifactStore`
- [ ] 视需要补充 `TraceStore`

第三层再提供：

- `SQLiteConversationStore`
- `SQLiteMemoryStore`
- 本地文件或对象存储 `ContextArtifactStore`
- 未来的向量存储实现

### 5.9 错误、取消与失败语义

第一层必须把错误和取消语义标准化，否则第三层任务系统与第四层协作调度会被迫自己猜测失败类型。

#### 第一层必须显式定义

- [ ] `LLMErrorCode`
- [ ] `ToolErrorCode`
- [ ] `RunErrorCode`
- [ ] `AbortReason`
- [ ] `TimeoutPolicy`
- [ ] `CancellationSignal` / `AbortSignal` 的接入方式

#### 设计要求

- [ ] 区分可重试与不可重试错误
- [ ] 区分“主动中止”“预算耗尽”“模型失败”“工具失败”“超时失败”
- [ ] `RunState` 必须能表达最终失败语义
- [ ] `AgentResult` 与异常抛出路径必须边界清晰

> 详细说明见：[`06_错误与取消语义模型.md`](./06_错误与取消语义模型.md)

### 5.10 执行轨迹与可观测边界

第一层要定义“记录什么”，第三层负责“写到哪里”。

#### 第一层负责

- [ ] `traceId`
- [ ] `StepTrace`
- [ ] `RunState`
- [ ] `AgentResult.steps`
- [ ] 每个 Hook 可见的运行态上下文

#### 第三层负责

- [ ] trace 持久化到数据库
- [ ] HTTP 查询接口
- [ ] 指标上报
- [ ] 系统日志与告警

- [ ] 每个 `run()` 都必须有唯一 `traceId`
- [ ] step 级信息必须可回放
- [ ] 执行结果必须能关联回具体 `traceId`

### 5.11 面向未来扩展的运行原语

第一层不直接实现“定时任务、编排模式、计划模式、投票模式”，但必须为它们提供统一执行原语。

#### 关键思想

未来各种高级能力，本质上都是对多个 `run()` 的组织。

##### 任务系统

- 负责“何时触发一次 run”
- 负责重试、超时、状态机
- 不重写第一层执行内核

##### 多 Agent 协作

- 每个 Agent 的一次执行各自运行在独立 Session 中
- 协作层传递的是任务描述、结果、中间产物、共享执行信息
- 不共享同一个 `ContextManager`

##### 编排模式

- 调度器按 DAG 或顺序组织多个 `run()`

##### 计划模式

- Planner 先生成计划
- 再由上层调度器发起多个后续 `run()`

##### 投票模式

- 并行发起多个 `run()`
- 收集多个 `AgentResult`
- 再由聚合器或另一个 Agent 决定最终结果

#### 第一层必须预留的能力

- [ ] `RunContext.metadata` 可透传上层调度信息
- [ ] `sessionType` 可扩展
- [ ] `AgentResult` 可被上层直接消费
- [ ] Hook 体系稳定
- [ ] 工具模型稳定
- [ ] 上下文压缩与外置策略可插拔

> 详细说明见：[`03_面向任务系统与多Agent扩展.md`](./03_面向任务系统与多Agent扩展.md)

### 7.1 首期完成标志

当以下闭环跑通时，可以认为第一层进入“可实现草案已验证”状态：

1. 创建 `AgentDefinition`
2. 创建或获取一个 `Session`
3. 通过 `SessionRegistry` 拿到 `SessionRuntime`
4. 使用 `ReActRunEngine` 发起一次 `run()`
5. 能完成至少一轮 `LLM -> Tool -> LLM` 或 `LLM -> Final`
6. 能返回带 `traceId`、`status`、`steps` 的 `AgentResult`
7. 能在取消、超时、工具失败时给出标准化状态与错误对象

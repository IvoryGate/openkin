# 01：运行时对象与执行引擎

> 配套主文档：`00_第一层技术方案总览.md`  
> 主题：统一运行时对象命名，彻底讲清 `Session`、`run()`、`step`、`RunEngine`、`RunState` 的关系  
> 本文档角色：可实现草案中的运行时 contract 与默认执行骨架

---

## 1. 为什么要重写这一层命名

第一层最容易让人混乱的地方，不是功能，而是“谁是对象，谁是流程，谁是状态”。

在前面的讨论中，最典型的混淆有这些：

- `Session` 和 `run()` 会不会是同一个东西
- `AgentRunner` 究竟是对象、类名还是一种执行模式
- 一次 `run()` 内部的状态到底要不要单独建模
- 生命周期 Hook 是不是“让 Session 自己执行流程”

为了解决这些问题，本轮重构统一采用下面这组命名：

- `AgentDefinition`
- `Session`
- `SessionRuntime`
- `RunEngine`
- `RunState`

其中最重要的变化是：

- 用 `RunState` 取代原先容易混淆的 `AgentRun`
- 用 `RunEngine` 明确表达“执行机制”，不再使用含糊的 `AgentRunner`

---

## 2. 五个核心概念

### 2.1 `AgentDefinition`

这是 Agent 的静态定义对象，回答的问题是：

- 它是谁
- 它默认怎么思考
- 它默认能用哪些能力

它通常包括：

- `id`
- `name`
- `systemPrompt`
- `llmProfile`
- `maxSteps`
- 默认工具集 / 默认插件集

它**不应包含**：

- 当前 messages
- 当前 stepIndex
- 当前 trace
- 当前 Session 的上下文

### 2.2 `Session`

这是上下文归属单位，回答的问题是：

- 这段对话历史属于谁
- 这次交互发生在哪个对话或任务容器中
- 这个上下文属于用户对话、内部子任务还是调度任务

一个 Session 可以存在很久，并承载多次交互。

### 2.3 `SessionRuntime`

这是 Session 在内存中的运行时资源集合，负责持有：

- `ContextManager`
- Session 级缓存
- 与当前 Session 绑定的辅助资源

一句话说：

- `Session` 是逻辑概念
- `SessionRuntime` 是内存中的真实运行对象

### 2.4 `RunEngine`

这是执行引擎，回答的问题是：

> 一次执行按什么机制运行？

它是“怎么跑”的机制抽象，不是“这次跑的状态对象”。

当前第一阶段的明确决策是：

- `RunEngine` 以接口形式存在
- 默认实现为 `ReActRunEngine`

未来可以扩展为：

- `PlanExecuteRunEngine`
- `SafeModeRunEngine`
- `ReviewRunEngine`

### 2.5 `RunState`

这是一次 `run()` 的运行态状态对象，回答的问题是：

> 当前这次执行进行到哪里了？

它通常应当持有：

- `traceId`
- `sessionId`
- 当前 `stepIndex`
- 本次输入
- 当前状态
- 已记录 `StepTrace[]`
- 最终结果
- 错误信息

注意：

`RunState` 会引用 `SessionRuntime`，但**不等于所有运行时对象的总和**。

---

## 3. 三层生命周期

### 3.1 Agent 级

Agent 级对象是可复用的、长生命周期的：

- `AgentDefinition`
- LLM provider 实例
- 默认配置

### 3.2 Session 级

Session 级对象是隔离上下文的：

- `Session`
- `SessionRuntime`
- `ContextManager`

### 3.3 Run 级

Run 级对象只属于一次执行：

- `RunState`
- `RunContext`
- `StepContext`
- 当前步骤计数
- 当前 step traces

### 3.4 为什么必须分层

如果不分层，最常见的错误是把 run 级状态误挂到 Agent 或 Session 上，例如：

- 当前 messages
- 当前 trace
- 当前 token 统计
- 当前工具执行中间结果

这样一旦多 Session 并发，同一个 Agent 就会发生状态污染。

---

## 4. `Session`、`run()`、`step` 的关系

### 4.1 一个 Session 可以有多次 run

用户在一个聊天窗口连续发送三次消息：

- 第一次消息 -> `run #1`
- 第二次消息 -> `run #2`
- 第三次消息 -> `run #3`

因此：

```text
Session
  ├─ Run #1
  ├─ Run #2
  └─ Run #3
```

### 4.2 一次 run 可以有多个 step

如果一次执行需要：

- 先思考
- 再调用搜索工具
- 再继续推理
- 最后输出答案

那么：

```text
Run #1
  ├─ Step 0
  ├─ Step 1
  ├─ Step 2
  └─ Step 3
```

### 4.3 三者关系总结

| 概念 | 意义 |
| --- | --- |
| `Session` | 一个持续存在的上下文容器 |
| `run()` | 在这个容器里发起的一次具体执行 |
| `step` | 这次执行内部的一轮推理循环 |

---

## 5. 为什么不让 Session 直接承担执行逻辑

直觉上很容易把设计写成：

```typescript
session.run(input)
```

这个形式作为**门面 API**可以接受，但内部仍不建议让 Session 自己承担完整执行逻辑。

### 原因一：职责不同

- Session 负责“归属和隔离”
- RunEngine 负责“流程控制”

### 原因二：Session 会变胖

如果让 Session 自己跑完整执行，就会往 Session 上不断塞入：

- `stepIndex`
- `traceId`
- llm invoke
- 工具循环
- Hook 调度

最终 Session 会从“容器”变成“大总管”。

### 原因三：不利于未来扩展不同执行策略

未来你很可能需要：

- 默认 ReAct 引擎
- 受限执行引擎
- 调试执行引擎
- 计划优先执行引擎

因此执行逻辑更适合抽象为：

```typescript
runEngine.run({
  agent: agentDefinition,
  runtime: sessionRuntime,
  input: agentRunInput,
})
```

而不是把这些策略写死在 Session 里。

---

## 6. 为什么 `RunEngine` 和 `RunState` 要同时存在

### 6.1 `RunEngine` 解决“怎么跑”

它是机制抽象，决定：

- 如何进入 step 循环
- 什么时候调用 LLM
- 什么时候跑工具
- 什么时候触发 Hook
- 什么时候结束

### 6.2 `RunState` 解决“这次跑到哪了”

它是状态容器，记录：

- 当前 `traceId`
- 当前 step
- 当前累计结果
- 当前错误状态

### 6.3 如果没有 `RunState`

如果不显式建模 `RunState`，这些状态就会散落到：

- `RunContext`
- `RunEngine` 的局部变量
- `SessionRuntime`
- 各种 Hook 临时状态

长期来看会越来越难维护。

---

## 7. 生命周期 Hook 到底是谁触发

### 7.1 触发者

生命周期 Hook 的触发者是 `RunEngine`。

原因：

- Hook 点位属于执行过程
- 只有执行引擎知道当前处于哪一个 step、哪一个阶段

### 7.2 执行者

真正执行 Hook 内业务逻辑的是各个插件。

例如：

- `ConversationHistoryPlugin`：加载历史
- `LongTermMemoryPlugin`：检索长期记忆
- `TraceLoggerPlugin`：记录轨迹
- `ToolPermissionGuard`：做权限校验

### 7.3 一句话总结

**`RunEngine` 负责在正确时机发出 Hook 调用，插件负责完成具体能力。**

---

## 8. 推荐对外入口与内部入口

### 8.1 对外入口

对上层暴露的入口可以很简单：

```typescript
agent.run(input)
```

### 8.2 内部入口

但内部应展开为：

```typescript
runEngine.run({
  agent: agentDefinition,
  runtime: sessionRuntime,
  input: agentRunInput,
})
```

这样可以同时兼顾：

- 对外 API 简洁
- 内部职责清晰
- 未来执行模式可替换

---

## 9. 推荐的最小接口关系

建议保留如下抽象：

```typescript
interface RunEngine {
  run(args: {
    agent: AgentDefinition
    runtime: SessionRuntime
    input: AgentRunInput
  }): Promise<AgentResult>
}

interface RunState {
  traceId: string
  sessionId: string
  stepIndex: number
  status: 'running' | 'completed' | 'aborted' | 'cancelled' | 'budget_exhausted' | 'failed'
  steps: StepTrace[]
  result?: AgentResult
  error?: RunError
}
```

这里最关键的是：

- `RunEngine` 是策略 / 机制
- `RunState` 是状态 / 结果容器

因此第一阶段虽然只有一个默认实现，但 contract 已经按“可替换执行引擎”来设计。

---

## 10. 对未来扩展的意义

只要第一层把 `Session / Run / Step / RunEngine / RunState` 这几个概念拆对，未来才能优雅支撑：

- 任务系统：调度多个独立 `run()`
- 编排模式：按顺序组织多个 `run()`
- 计划模式：Planner 先跑一次，再调度后续多个 `run()`
- 投票模式：并行发起多个 `run()`，再汇总结果
- 多 Agent 协作：每个 Agent 各自独立执行，协作层只传递任务和结果

换句话说：

**第一层不是直接实现这些高级模式，而是提供所有高级模式都会依赖的单次执行原语。**

---

## 11. 最终建议

建议将第一层运行时对象正式固定为以下五类：

1. `AgentDefinition`
2. `Session`
3. `SessionRuntime`
4. `RunEngine`
5. `RunState`

并在所有后续文档中遵守以下规则：

- 不再把 `Session` 和 `run()` 混用
- 不再把 `ContextManager` 挂在 Agent 上
- 不再把执行引擎和运行态状态对象混成一个概念
- 所有 Hook 都视为“执行过程中的扩展点”

---

## 12. 可实现草案：推荐的最小类型定义

如果第一层准备开始编码，建议先冻结以下最小类型：

```typescript
interface AgentRunInput {
  message: Message
  attachments?: MessagePart[]
  metadata?: Record<string, unknown>
}

interface RunOptions {
  abortSignal?: AbortSignal
  timeoutMs?: number
  maxSteps?: number
  maxToolCalls?: number
  maxPromptTokens?: number
}

interface RunArgs {
  agent: AgentDefinition
  runtime: SessionRuntime
  input: AgentRunInput
  options?: RunOptions
}

interface StepTrace {
  stepIndex: number
  promptTokens?: number
  completionTokens?: number
  toolCalls?: ToolCall[]
  toolResults?: ToolResult[]
  finishReason?: string
}
```

这组类型冻结后，`ReActRunEngine`、工具运行时、错误模型才能稳定衔接。

## 13. 可实现草案：默认 `ReActRunEngine` 骨架

建议默认实现严格按下面的职责顺序组织：

```typescript
class ReActRunEngine implements RunEngine {
  async run(args: RunArgs): Promise<AgentResult> {
    const state = this.createRunState(args)
    await this.runHooks('onRunStart', state)

    while (this.canContinue(state, args.options)) {
      await this.runHooks('onStepStart', state)
      const snapshot = await args.runtime.contextManager.buildSnapshot(state)
      const tools = await args.runtime.toolRuntime.getRuntimeView(state)
      const llmOutput = await args.runtime.llm.invoke({ snapshot, tools })

      if (llmOutput.toolCalls?.length) {
        const toolResults = await this.executeTools(llmOutput.toolCalls, state, tools)
        await args.runtime.contextManager.applyToolResults(toolResults, state)
        await this.runHooks('onStepEnd', state)
        continue
      }

      return await this.finishRun(state, llmOutput)
    }

    return await this.finishByBudgetOrAbort(state)
  }
}
```

这里最重要的是：

- `RunEngine` 控制流程
- `SessionRuntime` 提供资源
- `ContextManager` 提供快照
- `ToolRuntime` 提供当前可见工具
- `RunState` 只记录本次执行状态

## 14. 推荐的实现落点

建议第一层运行时最小先拆成这些文件：

- `agent/types.ts`：`AgentRunInput`、`RunOptions`、`AgentResult`
- `agent/run-engine.ts`：`RunEngine` 接口
- `agent/react-run-engine.ts`：默认 `ReActRunEngine`
- `agent/run-state.ts`：`RunState`、`StepTrace`
- `session/runtime.ts`：`SessionRuntime`

只要先把这些文件落稳，后面的 Hook、Context、Tool、Memory 都可以围绕它们补进来，而不用重写执行主干。
